# 2026-03-27 — Desktop + WASM dual-target pattern, launcher, dialogs

## cosmix-mond: headless system monitor daemon

Split cosmix-mon into two binaries following the daemon + GUI client pattern:

- **cosmix-mond** — headless daemon, registers as "mon" on hub, serves `mon.status` and `mon.processes` via AMP. Uses `sysinfo` crate for system metrics (CPU, memory, disk, load, processes).
- **cosmix-mon** — refactored to pure GUI client. No local sysinfo gathering — queries mond via hub. Builds as both desktop (native) and WASM (browser).

Deployed cosmix-mond to all 4 mesh nodes (cachyos, mko, gcwg, mmc) as systemd services. Verified cross-node monitoring: cachyos GUI querying all 4 nodes via mesh bridge.

## WASM dual-target infrastructure

Made the dependency chain WASM-safe to enable Dioxus apps building for both desktop and browser:

- **cosmix-port**: gated Unix socket code behind `cfg(feature = "native")`, split into `native_port.rs` module. tokio/libc now optional deps.
- **cosmix-client**: split into `native.rs` (tokio-tungstenite) and `web.rs` (gloo-net) with platform-conditional compilation. Added `connect_anonymous()` for GUI clients.
- **cosmix-web**: added `/ws` WebSocket proxy route so WASM apps can reach the hub through HTTPS. No CORS issues.
- **cosmix-mon**: added `web` feature flag and `Dioxus.toml` following cosmix-mail pattern.

This establishes the reusable pattern: daemon serves data via hub, GUI client works as both native desktop and browser WASM.

## cosmix-hub: mesh bridge response routing fix

Fixed a bug where responses to mesh-bridged requests were lost. The remote hub would route the response by `to` header (e.g. "test") which didn't exist locally. Added `PendingResponses` map — when routing a message with an `id` to a local service, the hub now tracks the caller's connection. Response messages with matching `id` route back to the caller instead of by `to` header.

## cosmix-edit: Open/Save As menu fix

Wired up muda menu event handler (`dioxus_desktop::use_muda_event_handler`) for File menu items. Open uses `rfd::AsyncFileDialog` file picker, Save As uses `rfd::AsyncFileDialog::save_file()`. Both menu items and keyboard shortcuts (Ctrl+O, Ctrl+S, Ctrl+Shift+S) now work.

## cosmix-launcher: system tray app launcher

New crate using `ksni` (StatusNotifierItem protocol) for COSMIC-compatible system tray. Discovers:
- Cosmix desktop apps from `~/.local/share/applications/` (Categories=Cosmix)
- Lua scripts from `~/.local/lua/*.lua`

Scripts have Run/Edit/Delete submenu. Run captures output and displays via cosmix-dialog. Delete confirms via cosmix-dialog. Menu rebuilds dynamically after actions.

Initial attempt with `tray-icon` crate (libappindicator) crashed COSMIC's status area by conflicting with the StatusNotifierWatcher D-Bus name. Switched to `ksni` which properly registers as an SNI client.

## cosmix-dialog: GUI dialog utility

New crate — drop-in replacement for zenity/kdialog, built with Dioxus. Five modes:

- `text-info` — scrollable text from stdin (script output display)
- `info` — info message with OK button
- `error` — error message with OK button
- `confirm` — yes/no, exit code 0=yes 1=no
- `input` — text input, prints to stdout

Used by cosmix-launcher for script output and delete confirmation. Usable standalone from bash, Lua, or any process.

## .desktop files and launcher integration

Created `.desktop` files for all GUI apps (cosmix-mon, cosmix-mail, cosmix-edit, cosmix-view, cosmix-launcher) in `~/.local/share/applications/` with `Categories=Cosmix`. Standard freedesktop icon names for now. All release binaries installed to `~/.local/bin/`.
