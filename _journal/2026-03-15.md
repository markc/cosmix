### Post-Deploy Setup: SSH and Bash

#### SSH key + sshd

openssh wasn't in the mesh image. Installed and enabled:
```bash
apk add openssh
rc-update add sshd default
rc-service sshd start
```

Added cachyos public key to `/root/.ssh/authorized_keys`.

#### Bash shell — three issues

1. **`/etc/passwd` already had `/bin/bash`** and bash was installed, but SSH sessions got `/bin/sh`. Alpine's busybox `login` doesn't set `$SHELL` from `/etc/passwd` the way glibc systems do.

2. **`$SHELL` env var fix:** Added `export SHELL=/bin/bash` to `/etc/profile` so the variable is set correctly in SSH sessions.

3. **SSH multiplexing stale socket:** Even after fixing the server side, SSH from cachyos kept reusing a stale multiplexed connection (via `~/.ssh/mux/192.168.2.162_22_root`). Had to remove the mux socket to force a fresh connection:
   ```bash
   rm ~/.ssh/mux/192.168.2.162_22_root
   ```

#### Post-deploy checklist for new Alpine mesh nodes

After launching a mesh CT (from mesh image or clone), run these steps from cachyos:

```bash
# 1. Install essentials (openssh, nano, rsync, bash already in image but verify)
incus exec NODE -- apk add openssh nano rsync bash

# 2. Fix root shell (mesh image defaults to /bin/sh)
incus exec NODE -- sed -i 's|root:x:0:0:root:/root:/bin/sh|root:x:0:0:root:/root:/bin/bash|' /etc/passwd

# 3. Add $SHELL export (Alpine busybox login doesn't set it from /etc/passwd)
incus exec NODE -- sh -c 'grep -q "export SHELL=/bin/bash" /etc/profile || echo "export SHELL=/bin/bash" >> /etc/profile'

# 4. Start sshd
incus exec NODE -- rc-update add sshd default
incus exec NODE -- rc-service sshd start

# 5. Sync shell config (creates .bash_profile, .bashrc, .myrc, .ssh structure)
rcm sync NODE

# 6. Remove stale SSH mux socket from cachyos (or first connection reuses old session)
rm -f ~/.ssh/mux/IP_22_root

# 7. Hush the Alpine MOTD
ssh NODE touch .hushlogin
```

#### Prevention for future mesh images

The mesh image build (`build-image.sh`) should bake in:
- `apk add openssh nano rsync bash`
- `rc-update add sshd default`
- `sed -i 's|/bin/sh$|/bin/bash|' /etc/passwd` (root shell)
- `export SHELL=/bin/bash` in `/etc/profile`
- `touch /root/.hushlogin`

This eliminates steps 1-3 and 7 from the post-deploy checklist.

### OpenRC Service Issues (Alpine + Unprivileged CT)

**Issue 9: `supervise-daemon --user` fails silently in unprivileged Proxmox CTs**

OpenRC's `supervisor=supervise-daemon` with `command_user="cosmic"` starts the supervisor but the child process dies immediately with no log output. The `--env` flag works, but the user switch fails silently in unprivileged containers.

**Fix:** Use `command_background=true` with `command="/bin/su"` and pass the real command as `command_args`:
```
command="/bin/su"
command_args="-s /bin/sh cosmic -c 'HOME=/home/cosmic XDG_RUNTIME_DIR=/run/user/1001 /usr/local/bin/cosmix-web'"
command_background=true
```

**Issue 10: `HOME` not set by supervise-daemon**

Even with `--env HOME=/home/cosmic`, the binary couldn't find its config at `~/.config/cosmix/web.toml` and fell back to a hardcoded default database (`markweb`). cosmix-web's default DB URL baked into the binary assumes the old markweb database.

**Fix:** Pass HOME explicitly in the su command args (see above).

**Issue 11: SMTP ports 25/587 in unprivileged CT**

`cosmix-jmap` failed with "Permission denied" trying to bind ports 25 and 587. Unprivileged Proxmox CTs can't bind below 1024 even with capabilities.

**Fix:** Changed jmap.toml to use high ports: `smtp_inbound = "0.0.0.0:2525"`, `smtp_submission = "0.0.0.0:2587"`.

**Issue 12: Memory requirement**

Initial 512 MB was insufficient — cosmix-embed needs ~320 MB for the nomic model. Raised to 1024 MB.

## Auto-Provisioning: Shortname + Hub Registration (PROTOTYPE COMPLETE)

Built and tested a fully automatic mesh node provisioning flow. New files:

- **`scripts/mesh-hub.sh`** — runs on gw router, manages WireGuard peer registry
- **`scripts/cosmixos-setup.sh`** — rewritten for automatic first-boot (replaces interactive version)

### How It Works

1. Node derives hostname from MAC: `n$(last 5 hex chars of first ethernet MAC)` → e.g. `nb165b`
2. Node generates WireGuard keys locally
3. Node SSHes to gw router: `mesh-hub.sh register <pubkey> <hostname>`
4. Hub assigns next free IP from 172.16.2.20-200, adds WireGuard peer, saves to registry
5. Node writes `wg0.conf`, brings up interface, starts cosmix services
6. Total time: ~5 seconds, zero human interaction

### End-to-End Test

```
incus launch cosmixos-mesh-20260314 mesh-test
incus exec mesh-test -- cosmixos-setup

=== CosmixOS Setup ===
Auto-assigned hostname: nb165b
Registering with mesh hub (gw)...
Hub assigned IP: 172.16.2.20
WireGuard interface up
=== Setup complete ===
```

Node appeared in mesh registry, pinged all peers, cosmix-web and cosmix-portd started automatically.

- **Permanent nodes**: human names (mko, gcwg, mmc) — have roles, domains, stable IPs
- **Auto-provisioned nodes**: shortnames (nb165b) — cloned workers, ephemeral, cattle not pets

## Incus `mesh` Profile

Created an Incus `mesh` profile on cachyos that bridges to `br0` (LAN) instead of the default `incusbr0` (NAT'd). This gives CTs real LAN IPs via DHCP from gw2 (192.168.2.0/24), matching how Proxmox CTs on the garage cluster get 192.168.1.0/24 IPs.

```bash
incus profile show mesh
# devices:
#   eth0:
#     nictype: bridged
#     parent: br0
#     type: nic
#   root:
#     path: /
#     pool: default
#     type: disk
```

Usage: `incus launch cosmixos-mesh-20260315 <name> --profile mesh`

### Why not `incusbr0` (NAT)?

The NAT'd `10.11.0.x` network had intermittent SSH connectivity to `192.168.1.1` (gw router) — TCP connections would sometimes time out despite ICMP ping working. Bridging to `br0` puts CTs directly on the LAN, eliminating NAT/conntrack issues and matching the Proxmox deployment model.

## `mesh-hub.sh` sed Fix (Base64 Pubkey)

The `sed` substitution in `mesh-hub.sh register` used `/` as delimiter. WireGuard pubkeys are base64 and contain `/` characters, which broke the JSON append. Fixed by switching to `|` delimiter.

## `cosmixos-setup.sh` SSH Quoting Fix

The SSH command in `cosmixos-setup.sh` had single quotes around `$PUBLIC_KEY` inside a double-quoted string, preventing variable expansion when pubkeys contained `+` or `/`. Removed the inner single quotes.

## Hub Topology Latency and Direct Peering

### The Problem

Current hub-and-spoke WireGuard topology routes ALL mesh traffic through the gw router. Pinging from cachyos to a local Incus CT on the same machine:

- `172.16.2.5` (cachyos itself): **0.05ms** — loopback, never leaves the machine
- `172.16.2.20` (nc06b1, local Incus CT): **~4ms** — two WireGuard hops through gw, TTL 63

4ms is fine for management traffic, but becomes a serious problem for bulk data transfers. A 2GB file between two nodes on the same LAN would travel: node → WireGuard → gw hub → WireGuard → destination. If the hub is a remote VPS (e.g. BinaryLane), that's 4GB of WAN traffic at WAN speeds — completely unacceptable.

### Three Options

**Option 1: Manual direct peers** — add a `wg2` interface between specific nodes when you know they'll exchange heavy traffic. Simple, doesn't scale, manual configuration per pair.

**Option 2: Automatic same-LAN direct peering** — `cosmixos-setup` already knows the node's LAN IP, and `mesh-hub.sh list` has every node's info. Nodes query the registry, find peers on the same subnet, and add direct WireGuard peers automatically. Hub only carries cross-subnet traffic. Fits the cosmix model well.

**Option 3: WireGuard mesh overlay** — nodes discover each other through a coordination server, then establish direct connections. The hub becomes a coordination point, not a relay.

### Mesh Overlay Tool Comparison

| | innernet | Nebula | NetBird |
|---|---|---|---|
| **Language** | **Rust** | Go | Go + TypeScript UI |
| **License** | MIT | MIT | BSD-3 + **AGPLv3** (mgmt) |
| **Self-hosted** | Yes, fully | Yes, fully | Yes (Docker-heavy) |
| **Discovery** | Coordination server | Lighthouse nodes | WebRTC ICE + STUN/TURN |
| **NAT traversal** | Manual endpoint override | UDP hole punching | Full ICE (STUN+TURN) |
| **Stars** | 5.4k | — | 23.5k |
| **Maintained** | Yes (v1.7.1, Nov 2025) | Yes (v1.10.3, Feb 2026) | Yes (v0.66.4, Mar 2026) |
| **Embeddable** | Yes (pure Rust, MIT) | Go FFI or sidecar | No (AGPLv3 core services) |

### Recommendation for Cosmix

**innernet is the best fit.** Pure Rust, MIT licensed, simple coordination model. Could be wrapped as a cosmix port or embedded directly into cosmix-daemon. The coordination server maps naturally to the mesh-hub concept.

**Nebula is second choice** — mature and battle-tested, but Go-based. Would run as a sidecar rather than embedded.

**NetBird is ruled out** — AGPLv3 on the management/signal/relay services creates licensing friction, Docker-heavy deployment conflicts with the cosmix philosophy.

### Decision

Not needed today — gw hub is local, 4ms is fine for current workloads. Becomes critical when:
- Hub moves to a remote VPS (BinaryLane)
- Heavy inter-node data transfers (LLM models, media files, database replication)
- Real-time agent coordination requiring sub-millisecond latency

Option 2 (automatic same-LAN peering) is the pragmatic first step. Option 3 (innernet integration) is the long-term architectural play.

## innernet Absorption Plan

Created `_doc/2026-03-15-innernet-absorption-plan.md` — comprehensive plan to absorb innernet's best components into cosmix-daemon as the WireGuard mesh foundation.

### Key Decisions

1. **Use `wireguard-control` crate as a dependency** — don't fork, just `cargo add`. It provides native netlink WireGuard management (~520 LOC, pure Rust, MIT).
2. **Skip innernet's CIDR ACLs, associations, invites, SQLite, hosts file, systemd** — we have our own solutions or don't need them.
3. **Absorb ~1,100 LOC** of innernet's ~4,000 LOC core: endpoint discovery, NAT traversal, peer state tracking, interface lifecycle.
4. **New `src/wg/` module** in cosmix-daemon: interface.rs, peers.rs, keys.rs, registry.rs (~1,280 new LOC).
5. **Replace SSH registration with AMP** — new nodes talk to hub daemon directly over WireGuard instead of SSHing to gw.
6. **Bootstrap via reserved IP** (172.16.2.254) — solves chicken-and-egg of needing WG to register but needing registration to get WG IP.
7. **No musl blockers** — wireguard-control is pure Rust + libc netlink. No C dependencies.

### Phases (target: 2 weeks)

| Phase | Days | Goal |
|---|---|---|
| 1 | 1 | wireguard-control dependency, WgManager, IPC commands |
| 2 | 1-2 | Peer registry in daemon (replaces mesh-hub.sh) |

### What This Enables

- Self-healing mesh (auto-reroute through hub on peer failure)
- Zero-touch scaling (boot CT → auto-join → direct peers)
- Scriptable infrastructure (`cosmix run mesh-report.lua`)
- The mesh itself becomes an ARexx port
- No external dependencies — pure Rust, single binary
