# 2026-03-14 — Mesh Node Update: Lessons Learned (mko)

Updated mko (172.16.2.210, web.kanary.org) to latest cosmix-web build. Documenting pitfalls for the remaining two nodes: gcwg (172.16.2.4) and mmc (172.16.2.9).

## Issue 1: PostgreSQL Table Permissions

**Symptom:** Login page shows "Internal error" instead of "Invalid email or password".

**Root cause:** The `users` table in the `cosmix` database was created by the `postgres` superuser (during initial schema setup), so it's owned by `postgres`. The cosmix-web service runs as the `cosmix` system user, which maps to the `cosmix` PostgreSQL role — which had no permissions on the table.

**Fix:**
```sql
sudo -u postgres psql -d cosmix -c "GRANT SELECT ON users TO cosmix;"
```

**Prevention checklist for other nodes:**
```sql
-- Run after any schema migration or table creation
sudo -u postgres psql -d cosmix -c "GRANT ALL ON ALL TABLES IN SCHEMA public TO cosmix;"
sudo -u postgres psql -d cosmix -c "ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT ALL ON TABLES TO cosmix;"
```

The second command ensures future tables created by `postgres` automatically grant access to `cosmix`.

## Issue 2: Stale Credentials in Secrets DB

The secrets database had multiple entries for kanary.org from different eras (old mail panel, old Laravel app, new cosmix-web). The correct credentials for cosmix-web are in the `users` table with bcrypt hashes — not necessarily matching any entry in `secrets.db`.

**Lesson:** After deploying cosmix-web to a node, verify the login works and update secrets.db if the password has changed.

## Issue 3: Wrong Password Entry Used

`admin@kanary.org` was a valid email in secrets.db but not a valid user in the cosmix-web `users` table. The actual user is `markc@kanary.org`. Always check:
```sql
sudo -u postgres psql -d cosmix -c "SELECT email FROM users;"
```

## Issue 4: Alpine Static Builds — Bind-Mount Fingerprint Mismatch

**Symptom:** After editing source files on the CachyOS host, `cargo build` inside the Alpine `cosmixos` CT reports "Finished in 0.4s" without recompiling — even after deleting fingerprints, deps, and the binary itself.

**Root cause:** The source tree is bind-mounted from the host into the CT (`incus config device add ... disk`). Files edited on the host carry host filesystem metadata (mtime, inode). Cargo inside the CT compares this against its own cached fingerprints and sees no change, because the metadata cross the mount boundary inconsistently.

**Fix:** Make any edit from *inside* the CT. Even a no-op `sed` is enough to give the file CT-native metadata:
```bash
incus exec cosmixos -- su - cosmic -c '
    cd ~/.gh/cosmix
    sed -i "1s/^/\/\/ rebuild\n/" crates/cosmix-jmap/src/main.rs
    sed -i "1d" crates/cosmix-jmap/src/main.rs
    cargo build --release --target x86_64-unknown-linux-musl -p cosmix-jmap
'
```

**Prevention:** When iterating on code that needs to be rebuilt inside the CT, either:
1. Edit files from inside the CT (e.g. `incus exec cosmixos -- vi ...`)
2. Use `git clone` inside the CT instead of bind-mounting (the desktop image uses this approach)
3. After host-side edits, touch the files from inside: `incus exec cosmixos -- touch path/to/file.rs`

## Issue 5: CARGO_TARGET_DIR and `build-alpine.sh`

**Symptom:** Running `cargo build` manually inside the CT puts binaries in `target/x86_64-unknown-linux-musl/release/`, but `build-alpine.sh` uses `target/alpine/x86_64-unknown-linux-musl/release/`.

**Root cause:** `build-alpine.sh` sets `CARGO_TARGET_DIR=target/alpine` to isolate Alpine builds from host glibc builds. Running cargo directly doesn't set this variable.

**Prevention:** When building manually inside the CT, either:
- Source the same env: `export CARGO_TARGET_DIR=target/alpine`
- Or just know the binary is at `target/x86_64-unknown-linux-musl/release/`

## Issue 6: cosmix-portd Needs `/run/user/$UID/`

**Symptom:** cosmix-portd reports "Permission denied" loading wrapper scripts, but files are readable.

**Root cause:** The error is misleading. It actually comes from `cosmix-port` trying to create a socket at `/run/user/{uid}/cosmix/ports/{name}.sock`. System users (created with `useradd -r`) have no login session, so `/run/user/{uid}/` doesn't exist. The error propagates up and gets attributed to the Lua file load.

**Fix:** Create the runtime directory and persist it:
```bash
mkdir -p /run/user/997/cosmix/ports
chown -R cosmix:cosmix /run/user/997

# Persist across reboots
cat > /etc/tmpfiles.d/cosmix.conf <<EOF
d /run/user/997 0700 cosmix cosmix -
d /run/user/997/cosmix 0755 cosmix cosmix -
d /run/user/997/cosmix/ports 0755 cosmix cosmix -
EOF
```

## Issue 7: rustls CryptoProvider Panic

**Symptom:** cosmix-jmap panics on startup: "Could not automatically determine the process-level CryptoProvider".

**Root cause:** Both `aws-lc-rs` (from sqlx) and `ring` (from hickory-resolver, mail-auth) features are enabled on rustls. With both present, rustls can't auto-detect which provider to use.

**Fix:** Call `install_default()` at the top of `main()`:
```rust
rustls::crypto::ring::default_provider()
    .install_default()
    .expect("Failed to install rustls crypto provider");
```
And add `features = ["ring"]` to the rustls dep in `Cargo.toml`.

## Architectural Breakthrough: Local-First + Federated Memory Sync

This is the defining insight for the cosmix mesh data architecture. Arrived at by working through the alternatives and ruling them out:

### What We Considered and Rejected

**Centralized PostgreSQL (single node):** Fewer resources (~630 MB saved per node), one backup target, simpler maintenance. But single point of failure — if mko goes down, every node loses auth and memory. And every write crosses WireGuard.

**PostgreSQL Streaming Replication (3-node):** Automatic data redundancy, local reads, hot standby for failover. But streaming replication is a data-center tool — it assumes reliable, always-on links. On a mesh that could span continents over WireGuard, WiFi, or flaky links: WAL accumulates when links drop, writes fail when the primary is unreachable, and automatic failover risks split-brain (two primaries accepting writes).

### What We're Building: Local-First + AMP Sync

**Every node owns its data. The mesh gossips knowledge when links are up.**

This is not a PostgreSQL feature — it's pure cosmix mesh architecture. PostgreSQL is just the local storage engine on each node. It has no idea other nodes exist. The sync logic runs through AMP messages over the mesh.

#### How It Works

Each memory entry carries origin metadata:

```sql
CREATE TABLE memories (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    key TEXT NOT NULL,
    content TEXT NOT NULL,
    embedding vector(768),
    origin_node TEXT NOT NULL,        -- "mko", "gcwg", "mmc"
    origin_id UUID NOT NULL,          -- original ID on source node
    visibility TEXT NOT NULL DEFAULT 'private',  -- private|mesh|node:NAME
    created_at TIMESTAMPTZ NOT NULL,
    updated_at TIMESTAMPTZ NOT NULL,
    synced_at TIMESTAMPTZ,            -- NULL = not yet synced
    UNIQUE(origin_node, origin_id)    -- prevents duplicate sync
);
```

When mesh links are up, nodes publish new memories as AMP messages. Receiving nodes UPSERT with last-write-wins conflict resolution:

```sql
ON CONFLICT (origin_node, origin_id) DO UPDATE
SET content = EXCLUDED.content, embedding = EXCLUDED.embedding,
    updated_at = EXCLUDED.updated_at
WHERE EXCLUDED.updated_at > memories.updated_at;
```

#### Graceful Degradation

| Mesh state | What happens |
|---|---|
| All links up | Sync runs, all nodes converge toward complete knowledge |
| One link drops for a day | Other nodes still sync. Disconnected node works with local + stale data |
| Total isolation | Every node works with what it has. No errors, no timeouts |
| Link comes back | Catches up — syncs everything it missed |

No promotion, no failover, no split brain. Nodes that can talk do talk. Nodes that can't just work.

#### Per-Node Privacy Control

Each node declares its sharing policy:

```toml
[memory.sync]
mode = "full"          # share everything, accept everything
# mode = "selective"   # share only memories marked 'mesh'
# mode = "isolated"    # share nothing, accept nothing
```

Per-memory visibility (`private`, `mesh`, `node:NAME`) gives fine-grained control. A node in `isolated` mode ignores sync commands entirely — no data leaves the node, guaranteed.

#### Why This Is the Right Answer

- **Autonomy:** Every node works independently, always. The mesh enhances but never gates functionality.
- **Privacy:** Each node controls what it shares. The memory port enforces the rules — no direct database access from the network.
- **Resilience:** No single point of failure. No split brain. Links come and go gracefully.
- **Simplicity:** No replication configuration, no WAL management, no failover promotion. The mesh already moves AMP messages — this is just another message type.
- **The ARexx model applied to storage:** Every node has a memory port. Scripts query whichever nodes are reachable. The data is local-first, federated second.

This is 100% justification for PostgreSQL + pgvector + cosmix-embed on every mesh node. The ~630 MB per node is the cost of full autonomy.

#### Resource Budget Per Node

| Component | RAM (idle) | Disk |
|---|---|---|
| PostgreSQL | 50 MB | ~100 MB |
| pgvector extension | negligible | negligible |
| cosmix-embed | 0 (socket-activated, exits after 60s) | 530 MB |
| **Total** | **50 MB** | **~630 MB** |

Cheap insurance for a node that never breaks when the network does.

## Architectural Breakthrough #2: DNS-Based Service Discovery

Every cosmix node runs cosmix-dns (hickory). SRV records enable zero-configuration service discovery across the mesh — no hardcoded IPs, no config file edits, no restarts when nodes join or leave.

### The _cosmix.amp Zone

```
; Per-node service registration
_postgres._tcp.mko._cosmix.amp.     SRV 0 0 5432  mko.amp.
_embed._tcp.mko._cosmix.amp.        SRV 0 0 0     mko.amp.
_web._tcp.mko._cosmix.amp.          SRV 0 0 443   mko.amp.
_jmap._tcp.mko._cosmix.amp.         SRV 0 0 8443  mko.amp.
_ollama._tcp.ollama2._cosmix.amp.   SRV 0 0 11434 ollama2.amp.
_ollama._tcp.ollama3._cosmix.amp.   SRV 0 0 11434 ollama3.amp.

; Mesh-wide queries: find ALL instances of a service
_postgres._tcp._cosmix.amp.  SRV 10 0 5432 mko.amp.
_postgres._tcp._cosmix.amp.  SRV 10 0 5432 gcwg.amp.
_postgres._tcp._cosmix.amp.  SRV 10 0 5432 mmc.amp.
_ollama._tcp._cosmix.amp.    SRV 10 0 11434 ollama2.amp.
_ollama._tcp._cosmix.amp.    SRV 10 0 11434 ollama3.amp.

; WireGuard IPs
mko.amp.      A 172.16.2.210
gcwg.amp.     A 172.16.2.4
mmc.amp.      A 172.16.2.9
ollama2.amp.  A 172.16.2.11
ollama3.amp.  A 172.16.2.12
```

### What This Enables

**Federated memory sync discovers peers from DNS:**
```lua
function discover_memory_peers()
    local all_pg = dns.query("_postgres._tcp._cosmix.amp", "SRV")
    return filter(all_pg, function(srv) return srv.host ~= self_host end)
end
```

**Ollama round-robin discovers inference nodes from DNS:**
```lua
function discover_ollama_nodes()
    return dns.query("_ollama._tcp._cosmix.amp", "SRV")
    -- Returns: [{host="ollama2.amp", port=11434}, {host="ollama3.amp", port=11434}]
end
```

**New node joins the mesh:**
1. Install cosmix, start cosmix-dns
2. Register SRV records for its services
3. Every other node discovers it on the next query
4. No config changes anywhere else

**Node goes down:**
- DNS TTL expires, SRV records go stale
- Health checks confirm unreachable
- Other nodes route around it automatically

### Integration with ARexx Port Discovery

This complements the existing port discovery (inotify on `/run/cosmix/ports/`). Local ports are discovered by filesystem, remote services by DNS. Same ARexx principle — everything is discoverable, nothing is hardcoded:

| Scope | Discovery mechanism |
|---|---|
| Local ports (Layer 3) | inotify on Unix socket directory |
| Local services | localhost SRV in cosmix-dns |
| Mesh services | `_service._tcp._cosmix.amp` SRV queries |
| Cross-mesh (future) | Federated DNS between mesh zones |

### Self-Registration

Each cosmix binary should register its own SRV records on startup and deregister on shutdown:

```rust
// In cosmix-web main()
dns.register("_web._tcp", node_name, 443)?;
// On shutdown:
dns.deregister("_web._tcp", node_name)?;
```

This makes the mesh fully self-organizing — start a service, it appears in DNS, every other node discovers it automatically.
