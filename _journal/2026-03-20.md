# 2026-03-20 — Firefox Unresponsive Window Investigation

## Session 1: cosmic-comp Surface Tracking Bug

### The Problem

Firefox main window periodically stops responding to mouse clicks. Tabs can't be opened, buttons don't respond. Restarting Firefox fixes it. This has been happening since at least 2026-03-15 (first journaled mention) and is still occurring today. Thunderbird has a similar issue (compose window opens tiny and empty).

### Why Firefox?

Firefox/Gecko apps are disproportionately affected because:

1. **COSMIC native apps bypass the buggy code path** — libcosmic/iced apps don't use `xdg_popup` Wayland protocol, so they never hit the popup tracking code in `xdg_shell/popup.rs`
2. **Firefox has its own Wayland popup implementation** — unlike GTK/Qt apps that use their toolkit's backend, Gecko rolled its own, and it's aggressive: URL bar autocomplete creates/destroys/repositions popup surfaces on every keystroke
3. **Popup repositioning is the trigger** — issue #1815 specifically mentions "Unable to re-configure repositioned popup", and Firefox constantly repositions its autocomplete popup as suggestions change (most apps create a popup at a fixed position then destroy it)
4. **GTK/Qt apps hit the same path less often** — they create fewer popups, don't reposition mid-life, and have simpler hierarchies

However — today's investigation revealed the popup tracking may not be the whole story (see below).

## Session 2: Further Testing

### killall cosmic-comp

Restarts entire COSMIC session — all apps killed, useless as a workaround.

### Toolbar Clicks as Reliable Trigger

**Any toolbar click** triggers the freeze — not just bookmarks. Testing confirmed:

1. Typed 3 URLs into the address bar using keyboard — all worked fine
2. Navigated YouTube, clicked around the page content — ~12 clicks worked normally
3. Pulled up a video successfully via page clicks
4. Clicked the **reload button on the toolbar** → immediately frozen

The pattern: **page content clicks work fine, toolbar clicks freeze input**. The bookmarks bar was a red herring as the specific trigger — it just happens to be the most natural first click. The real trigger is any Firefox toolbar interaction, which creates native Wayland popup/menu surfaces (dropdown menus, button tooltips, URL autocomplete). Page content clicks go through Firefox's internal rendering and never create Wayland popups.

This further confirms the cosmic-comp `xdg_shell/popup.rs` surface tracking bug — toolbar widgets create popup surfaces that cosmic-comp loses track of, locking the input grab.

### cosmic-settings-daemon Spin Bug: Root Cause Found

**Previously hypothesized:** D-Bus feedback loop between cosmic-osd and cosmic-settings-daemon.

**Actual cause:** PulseAudio mainloop busy-spin in cosmic-settings-daemon.

#### Evidence

Ran `perf record -g -p <pid> -- sleep 3` while the process was at 92% CPU. 100% of samples landed in:

```
pa_mainloop_poll → ppoll / pa_read
```

Meanwhile `strace` showed **zero syscalls** in 5 seconds — the spin is entirely in userspace. The `ppoll` calls return immediately every time (zero timeout or a broken/always-ready fd), so `pa_mainloop_iterate` never blocks.

#### Audio Stack (confirmed correct)

| Component | Status |
|-----------|--------|
| PipeWire 1.6.2 | Running (PID 3638) |
| pipewire-pulse 1.6.2 | Running (PID 3642), provides PulseAudio API compat |
| PulseAudio daemon | Not installed, not running — correct |
| libpulse (client lib) | Linked by cosmic-settings-daemon — normal |

The audio stack setup is correct. cosmic-settings-daemon uses `libpulse` to talk to `pipewire-pulse` for volume/audio settings. The bug is in how cosmic-settings-daemon drives the `pa_mainloop` — likely calling `pa_mainloop_iterate(0)` (non-blocking) in a tight loop without any sleep, or receiving a fd from pipewire-pulse that's perpetually in a ready state.

cosmic-osd likely has the same bug (also showed `pa_mainloop_poll` in earlier traces before it exited during investigation).

#### The Complete Bug Chain

1. **Boot** → cosmic-settings-daemon and pipewire-pulse both start at the same time (09:23:50)
2. **Race condition** → pulse thread connects before pipewire-pulse is fully ready, context never reaches `Ready` state
3. **Busy spin** → `pa_mainloop_iterate(false)` loops at 100% CPU with no sleep/backoff/timeout
4. **Wedges pipewire-pulse** → the relentless non-blocking connection attempts (or held memfd/socket) make pipewire-pulse unresponsive to ALL clients — `pactl info` returns "Connection failure: Timeout"
5. **Cascade** → cosmic-osd also spins (same code path via cosmic-settings-subscriptions). Every app that needs audio is blocked — Firefox windows that depend on audio/media pipeline stall and become invisible

#### Proof: PipeWire Restart Fixed Everything

```bash
systemctl --user restart pipewire.service pipewire-pulse.service wireplumber.service
```

After restarting the PipeWire stack:
- `pactl info` responded immediately
- **Firefox suddenly appeared and resumed playing the last video, with audio**
- The "invisible Firefox" and "Firefox not accepting clicks" were downstream symptoms of pipewire-pulse being wedged — not (only) the cosmic-comp popup tracking bug

This means some or all of the Firefox issues from Session 1 may have been caused by the audio pipeline being broken, not cosmic-comp surface tracking. The popup tracking bugs still exist upstream, but the primary trigger on this system was the pipewire-pulse wedge.

#### Two Upstream Bugs

1. **cosmic-settings-daemon** (pop-os/cosmic-settings-daemon) — `cosmic-settings-subscriptions/src/pulse.rs` connection loop uses `iterate(false)` (non-blocking) with no sleep/backoff/timeout. If the context never reaches `Ready`, it spins forever. Fix: add a sleep between iterations, or use `iterate(true)`, or add a retry limit with exponential backoff.
2. **pipewire-pulse** (PipeWire) — should not become unresponsive to all clients because one client is hammering it with non-blocking connection attempts. Needs connection rate limiting or isolation between clients.

#### Workaround

Restart the entire PipeWire stack:
```bash
systemctl --user restart pipewire.service pipewire-pulse.service wireplumber.service
```

This recovers everything — audio, Firefox visibility, the lot. `kill -STOP` on the spinning processes alone is insufficient because pipewire-pulse is already wedged by that point.

## Session 6: PipeWire Boot Fix Workaround (Enabled)

### Context

Session 5 patched `cosmic-settings-daemon` with `iterate(true)` to fix the busy-spin, and disabled the `pipewire-fix-race.service` workaround to test the code fix in isolation. The code fix stopped the CPU spin (0% at T+10s and T+20s), but PipeWire itself still deadlocks on boot — `pactl` times out, no audio devices appear in the COSMIC audio settings panel.

### The Cascade (Confirmed)

1. **PipeWire deadlocks on boot** — race condition, sockets exist but don't respond
2. `cosmic-settings-daemon` spins (fixed by our patch, but PipeWire is still stuck)
3. `cosmic-osd` spins at 100% CPU retrying volume/device notifications
4. Audio panel shows no devices (`pactl info` → "Connection failure: Timeout")
5. Firefox/media apps stall waiting for audio pipeline

### Fix: Improved `pipewire-fix-race.service`

Re-enabled and improved the systemd user service. Previous version blindly restarted PipeWire on every login. New version checks first:

```ini
[Unit]
Description=Fix PipeWire boot deadlock that cascades to COSMIC audio daemons
After=graphical-session.target pipewire-pulse.service wireplumber.service

[Service]
Type=oneshot
ExecStart=/bin/bash -c '\
  sleep 8; \
  if ! timeout 5 pactl info >/dev/null 2>&1; then \
    logger -t pipewire-fix "PipeWire unresponsive after boot, restarting stack"; \
    systemctl --user restart pipewire.service pipewire-pulse.service wireplumber.service; \
    sleep 2; \
    pkill -x cosmic-osd; \
  else \
    logger -t pipewire-fix "PipeWire responsive, no fix needed"; \
  fi'

[Install]
WantedBy=graphical-session.target
```

**Behaviour:**
1. Wait 8s for PipeWire to settle
2. Test with `timeout 5 pactl info` — if responsive, log and exit (no-op)
3. If stuck — restart full PipeWire stack, wait 2s, kill `cosmic-osd` so it respawns clean
4. Logs to journal: `journalctl --user -t pipewire-fix`

| `iterate(true)` patch in cosmic-settings-daemon | CPU spin bug | Installed, working |

## Session 9: cosmixos Alpine CT — Audio & Timezone Fixes

Running Claude Code inside the cosmixos Incus CT (Alpine Edge, musl libc). These fixes are CT-side only — the CachyOS host (glibc) was already fixed in Sessions 2–7.

### Audio: Restoring cosmic-settings-daemon & cosmic-osd

**Problem:** The COSMIC audio applet showed no devices and volume control was non-functional. `cosmic-settings-daemon` and `cosmic-osd` were not running.

**Root cause:** Both `/usr/bin/cosmic-settings-daemon` and `/usr/bin/cosmic-osd` had been replaced with stub scripts during an earlier session to prevent the 100% CPU busy-spin race condition (the same bug fixed on the host side in Session 5 with `iterate(true)`):

```sh
#!/bin/sh
exec sleep infinity
```

cosmic-session was dutifully "starting" them, but they were just sleeping forever — no PulseAudio connection, no audio device management.

**Why it's safe to restore now:** The CT doesn't run its own PipeWire. Instead, the host's PipeWire sockets are bind-mounted into the CT:

```
/run/user/1001/pipewire-0        → /mnt/.host-run/pipewire-0 (host socket)
/run/user/1001/pulse/native      → /mnt/.host-run/pulse/native (host socket)
```

The host PipeWire is always running before the CT's COSMIC session starts (via `start-cosmixos` script), so the race condition that causes the busy-spin — daemon starts before pipewire-pulse is ready — cannot occur in this setup.

**Verification:** `speaker-test -c 2 -t sine -l 1` produced audible output through the Audient iD4 before restoring the daemons, confirming the audio pipeline (CT → host PipeWire socket → hardware) was functional. `pactl info` also connected successfully, reporting `Server Name: PulseAudio (on PipeWire 1.6.2)` with `Host Name: cachyos`.

**Fix:**

```bash
sudo apk fix --force cosmic-settings-daemon cosmic-osd   # reinstall real ELF binaries
kill 516 524                                              # kill sleep-infinity stubs
# cosmic-session auto-respawned with real binaries
```

**Result:** Both daemons running at ~0.1% CPU (no spin). Audio applet shows devices and volume controls.

Long-term fix: build the patched version (`iterate(true)`) for musl from the `markc/cosmic-settings` fork, or wait for upstream PR #1919 to be merged and packaged.

## Session 10: NFS Mount Visibility in cosmixos CT

### Problem

The host NFS mount at `/home/cosmix/Downloads` (from `192.168.2.25:/rpool/media`, 918GB) was invisible inside the cosmixos Incus CT, despite `/home/cosmix` being bind-mounted into the CT via the `home-cosmix` disk device.

Inside the CT, `/home/cosmix/Downloads` existed but was empty (2 entries: `.` and `..`) — showing the underlying ZFS directory, not the NFS overlay.

### Root Cause

The `home-cosmix` disk device uses `shift: true` (UID/GID idmapping), which creates an isolated mount namespace. This prevents submount propagation — even with `propagation=shared` set on the device, NFS mounts on the host that sit underneath the bind-mounted path are not visible inside the container.

This is a fundamental Linux mount namespace limitation: bind mounts capture the filesystem at the mount point, not any submounts layered on top. The `shift` option makes it worse by adding an additional namespace layer.

### Fix

Added the NFS mount as a separate Incus disk device, so Incus mounts it directly into the CT:

```bash
incus config device add cosmixos nfs-downloads disk \
  source=/home/cosmix/Downloads \
  path=/home/cosmix/Downloads \
  shift=true
```

No container restart was required — Incus hot-attached the device.

### Verification

```
cosmixos $ df -h /home/cosmix/Downloads
Filesystem                Size      Used Available Use% Mounted on
192.168.2.25:/rpool/media 917.8G    18.8G    899.1G   2% /home/cosmix/Downloads

cosmixos $ ls /home/cosmix/Downloads/
CLAUDE.md  movies  music  photos  shows  videos
```

### Final Device Configuration

```
home-cosmix:
  path: /home/cosmix
  propagation: shared
  shift: "true"
  source: /home/cosmix
  type: disk
nfs-downloads:
  path: /home/cosmix/Downloads
  shift: "true"
  source: /home/cosmix/Downloads
  type: disk
```

Both devices are persistent in the Incus database — survives container and host reboots. The only dependency is that the host NFS mount (`192.168.2.25:/rpool/media` on `/home/cosmix/Downloads`) must be up before the container starts.

## Session 11: Ctrl-C Kills COSMIC Session — Fix Re-applied

### Problem

Pressing Ctrl-C in any cosmic-term shell inside the cosmixos CT (e.g., to stop `ping`) kills the entire COSMIC desktop session, dropping back to the host greeter. This was previously fixed around 2026-03-18/19 but the fix was never journaled and was lost — likely during the `cosmic` → `cosmix` user rename.

### Root Cause

The `start-cosmixos` script launches cosmic-session via:

```
incus exec cosmixos -- su - cosmix -c 'exec dbus-run-session -- cosmic-session'
```

This creates a signal chain: host TTY → `incus exec` → `su` → `dbus-run-session` → `cosmic-session` → `cosmic-comp` → `cosmic-term` → shell. When Ctrl-C is pressed in cosmic-term, SIGINT propagates up through this entire chain because:

1. The host-side script has no SIGINT protection (`set -ex` makes it worse)
2. `cosmic-session` shares the controlling terminal's session/process group via the `incus exec` ancestry

### Fix (in `scripts/start-cosmixos`)

Two changes to the cosmic-session launch block:

1. **`trap '' INT`** on the host side before `incus exec` — prevents the host-side process from dying on SIGINT
2. **`setsid`** inside the CT — `exec setsid dbus-run-session -- cosmic-session` gives cosmic-session its own process session, fully isolating it from the controlling terminal's signal routing

After cosmic-session exits, `trap - INT` restores default signal handling for the cleanup phase.

### Verification

Confirmed no trap/stty fix existed anywhere in the CT:
- No `trap` or `stty` in `/etc/profile`, `/etc/profile.d/`, `~/.bashrc`, `~/.profile`, or `~/.bash_profile`
- The devpts mount is correct (`ptmxmode=666`) — PTY isolation itself is fine
- The issue was purely signal propagation through the `incus exec` process chain

## Session 12: PipeWire Auto-Start on Login/Logout

### Context

Inside the cosmixos Incus CT (Alpine Edge, musl), PipeWire needs to run natively for audio — the host socket passthrough approach from Session 9 was replaced by running the full PipeWire stack inside the CT. But PipeWire wasn't starting automatically, requiring manual `pipewire &; wireplumber &; pipewire-pulse &` every login.

Alpine uses OpenRC, not systemd — there are no `systemctl --user` services available.

### Fix: `~/.profile` auto-start

Replaced the old host-socket symlink block in `~/.profile` with native PipeWire startup:

```sh
# Start PipeWire audio stack (native in CT, ALSA devices passed through from host)
if ! pgrep -u "$(id -u)" pipewire >/dev/null 2>&1; then
    pipewire &
    sleep 0.3
    wireplumber &
    sleep 0.3
    pipewire-pulse &
fi
```

The `pgrep` guard ensures it's idempotent — multiple SSH sessions won't spawn duplicate PipeWire instances.

### Fix: `~/.bash_logout` cleanup

Added PipeWire teardown to `~/.bash_logout` to kill the audio stack on logout:

```sh
pkill -u "$(id -u)" pipewire-pulse 2>/dev/null
pkill -u "$(id -u)" wireplumber 2>/dev/null
pkill -u "$(id -u)" pipewire 2>/dev/null
```

This prevents stale PipeWire processes lingering after logout, which would conflict with the host's own PipeWire when working in the CachyOS glibc environment.

### What was removed

The old `~/.profile` block that symlinked host PipeWire sockets (`/mnt/.host-run/pipewire-0` → `/run/user/1001/pipewire-0`) is gone. That approach worked for passthrough but we're now running PipeWire natively in the CT with ALSA devices passed through from the host.
