## Ollama Infrastructure

### Qwen3 nothink investigation
- `/no_think` in system prompt does NOT work — Qwen3 ignores it
- `/nothink` (no underscore) is the correct Qwen3 directive, but still unreliable
- The **only reliable method** is `--think=false` CLI flag or `"think": false` in API JSON
- Cannot be baked into a Modelfile — it's an API/CLI-only parameter
- `qwen3.5:9b` respects `think: false` properly; `qwen3:30b-a3b` leaks thinking into content regardless

### Ollama node setup
- **ollama3** (pve3, VMID 130, 192.168.2.133) — was inaccessible from cachyos due to missing SSH authorized_key (not MAC/IP issue). Fixed.
- **ollama2** (pve2, VMID 101, 192.168.2.132) — already existed, was stopped. Started, fixed subgid mapping, configured.
- Both CTs had `OLLAMA_VULKAN=1` which offloaded layers to broken Intel Iris Xe Vulkan — caused inference to hang
- **Fix for all 3 nodes:** removed Vulkan, set `OLLAMA_NUM_GPU=0` in systemd override
- Removed GPU passthrough (`/dev/dri/renderD128`) from both CT configs — not needed for CPU-only
- cachyos also had `OLLAMA_VULKAN=1` as a ticking time bomb — fixed

## Phase 8: AI Agent Scripts

Built four Lua scripts implementing the agent architecture:

### scripts/agent.lua — Sequential agent with tools
- Single agent loop: task → model → tool call → execute → feed back → repeat
- 6 tools: list_dir, read_file, write_file, run_command, search_files, grep
- Tools execute locally on cachyos regardless of which node runs inference
- Round-robins across all 3 nodes per turn, with automatic failover
- Can pin to single node with `AGENT_NODE=ollama2`

### scripts/swarm.lua — Parallel agents, no tools
- Planner decomposes task into N subtasks (one per node)
- Workers run in parallel via background curl processes
- Synthesiser combines results into final answer
- 5 LLM calls total: planner + 3 workers + synthesiser
- Good for research/comparison tasks

### scripts/haiku-race.lua — Parallel race, same prompt
- Fires identical prompt to all nodes simultaneously
- Prints results as they arrive with timing
- Shows per-node tokens, tok/s, wall-clock time, and aggregate stats

### scripts/crew.lua + crew-worker.lua — Parallel agents WITH tools
- The full vision: multiple agent loops with tools running simultaneously
- crew.lua = supervisor (plans, launches workers, streams logs, synthesises)
- crew-worker.lua = standalone agent loop with tools, reads config from JSON file
- Each worker runs as a background Lua process on its own node
- Live log streaming shows tool calls as they happen across all workers
- Workers have 5-turn limit to prevent runaway loops

### Architecture pattern
```
crew.lua (supervisor)
  → Planner (cachyos): "split into 3 subtasks"
  → Worker 1 (cachyos): agent loop + tools
  → Worker 2 (ollama2): agent loop + tools    } parallel
  → Worker 3 (ollama3): agent loop + tools
  → Synthesiser (cachyos): "combine results"
```

Brain on any node, hands on cachyos. Models are stateless HTTP endpoints.
Lua does all orchestration — the ARexx model realised.

## CosmixOS: Nested COSMIC Desktop — IT LIVES!

The white whale has been harpooned. A full COSMIC desktop session running inside
an Alpine Edge LXC container, rendered as a native Wayland window on the host
desktop. No RDP. No VNC. No network encoding. Pure GPU-accelerated nested
Wayland glory.

### What we built

An Alpine Edge container (`cosmixos` at 10.11.0.117) running the complete
COSMIC 1.0.8 desktop stack nested inside the host's COSMIC session via
cosmic-comp's Winit backend. The nested compositor connects to the host's
Wayland socket, gets its own GPU-accelerated rendering pipeline, and appears
as a regular window you can move, resize, and alt-tab to.

### The two-socket architecture (the key insight)

This is what makes it all work:

```
Host compositor (cosmic-comp DRM) → /mnt/wayland/wayland-1
  └── Nested compositor (cosmic-comp Winit) connects as client to wayland-1
        └── Creates wayland-2 for ITS clients (panel, term, files, etc.)
```

The initial attempts used `cosmic-session` which launched everything against
`wayland-1` — so panel applets, the launcher, and workspace controls were
talking to the HOST compositor, not the nested one. That's why actions inside
the nested window were activating things on the parent desktop. Delightfully
confusing until we figured it out.

### The launch architecture

Two scripts, because nested shell quoting is a circle of hell:

**Host:** `~/.local/bin/cosmixos` — starts container, kicks off inner script
**Container:** `~/start-cosmic.sh` — the real orchestrator:
1. cosmic-comp starts against `wayland-1` (host socket)
2. Waits for `wayland-2` socket to appear (nested compositor ready)
3. Sets 225% scale via `cosmic-randr` (4K display)
4. Launches session components against `wayland-2`
5. Starts clipboard bridge daemon
6. Graceful shutdown with SIGTERM + 2s drain on exit

All stdout/stderr suppressed via fd 3 trick — only "CosmixOS ready" gets through.

### Problems solved

| Problem | Solution |
|---------|----------|
| Broken pipe spam on startup | Session clients were connecting to host socket, not nested — fixed by two-socket architecture |
| Tiny UI elements at 4K | `cosmic-randr mode WINIT-0 2560 1350 --scale 2.25` after compositor starts |
| Keyboard stops working mid-session | Removed `cosmic-settings-daemon` — its global shortcut grabs conflict with host compositor |
| No clipboard sharing | Bash polling bridge syncs `wl-paste`/`wl-copy` between `wayland-1` and `wayland-2` every 500ms |
| Noisy startup (X11 warnings, EGL, xcursor) | Suppressed entire `dbus-run-session` block via `3>&1 &>/dev/null` |
| Broken pipe spam on exit | Graceful shutdown: `kill -TERM 0` + 2s sleep before shell exits |
| `$!` not captured (busybox sh) | Moved script inside container as `~/start-cosmic.sh` using bash, not sh |
| No sudo/ping | Installed sudo, passwordless for cosmic user, suid on ping |

### Container setup summary

- **Base:** Alpine Edge, unprivileged LXC with `raw.idmap: both 1001 1001`
- **GPU:** Full `/dev/dri` passthrough (Intel Arc xe, Mesa 26.0.2)
- **Wayland:** Host's `/run/user/1001` mounted at `/mnt/wayland` with `shift=true`
- **User:** `cosmic` (UID 1001), groups: render, video, input
- **Packages:** Full COSMIC 1.0.8 stack (22 cosmic-* packages), Pop cursor theme
- **Not running:** `cosmic-settings-daemon`, `cosmic-launcher` (shortcut conflicts)

### What this means for CosmixOS

This is the foundation for the CosmixOS vision — an isolated, reproducible COSMIC
environment that can be snapshotted, cloned, migrated between hosts, and used as
a development sandbox. The container shares the host GPU but has its own D-Bus,
its own filesystem, its own package manager. You can `apk upgrade` fearlessly
inside it without touching the host.

It's also proof that cosmic-comp's Winit backend works beautifully for nesting.
The rendering is indistinguishable from native — because it IS native GPU
rendering, just composited as a Wayland client window.
