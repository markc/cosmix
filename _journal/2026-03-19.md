## Session 1: Claude Code Inside cosmixos CT

### Context

First Claude Code session running directly inside the cosmixos Alpine CT (not from the CachyOS host). The CT is an Incus privileged container running Alpine Edge with the full COSMIC desktop — achieved "first light" on 2026-03-17 (see that journal entry, Sessions 4–9).

### Immediate finding: bash tool broken

Claude Code's Bash tool is non-functional inside this CT. The `claude` binary bundles its own bash and readline libraries (`/host-lib/libreadline.so.8`, `/host-lib/libncursesw.so.6`) which are glibc-linked. Alpine uses musl libc, so every glibc symbol (`__snprintf_chk`, `__vfprintf_chk`, `__mbrlen`, etc.) fails to resolve. Every shell command exits 127.

This means Claude Code in this CT can:
- Read files (Read tool — works, uses Node.js not bash)
- Write/edit files (Write/Edit tools — same)
- Search files (Glob/Grep — same)
- Launch subagents

But **cannot**:
- Run any bash commands (cargo build, git, system commands)
- Execute scripts
- Check system state (hostname, services, processes)

### Workarounds to investigate

1. **Install glibc compatibility layer** — Alpine has `gcompat` package (`apk add gcompat`) which provides a glibc shim on musl. May resolve the symbol errors.
2. **Symlink Alpine's native bash** — if Claude Code can be told to use `/usr/bin/bash` (Alpine's musl-linked bash) instead of its bundled one.
3. **Set `LD_LIBRARY_PATH`** — may be able to override the host-lib path to use musl equivalents.
4. **Run Claude Code from host** — continue using `incus exec cosmixos -- ...` from the CachyOS host for commands that need bash, use CT-local Claude Code for file editing only.

### Ctrl-C kills entire COSMIC session

**Critical usability bug.** Pressing Ctrl-C anywhere in cosmic-term drops the entire COSMIC session back to the greeter. This is NOT a Claude Code issue — happens in any shell inside cosmic-term.

**Likely cause:** PTY isolation is broken. `/dev/pts` may not be properly mounted as `devpts` in the CT. Without proper PTY isolation, Ctrl-C sends SIGINT to the session's controlling terminal (cosmic-session's process group) instead of just the foreground process in the terminal emulator's PTY.

**Diagnosis needed (from host side):**
```bash
# Check devpts mount inside CT
incus exec cosmixos -- sh -c 'mount | grep pts'
incus exec cosmixos -- sh -c 'ls -la /dev/pts/'

# Fix if devpts not mounted properly
incus exec cosmixos -- sh -c 'mount -t devpts devpts /dev/pts -o newinstance,ptmxmode=0666'

# Check if cosmic-comp survives a Ctrl-C (run from host after Ctrl-C in CT)
incus exec cosmixos -- pgrep -a cosmic-comp
```

**Alternative theory:** cosmic-comp may not have a SIGINT handler. On a normal desktop it never receives SIGINT, but in the container the signal routing may be different. If cosmic-comp catches SIGINT and exits, greetd restarts the greeter.

**Workaround:** `trap '' INT` in shell profile, or `stty intr undef` to disable Ctrl-C entirely in the terminal.

### COSMIC spin bug: cosmic-osd and cosmic-settings-daemon

Two COSMIC desktop processes on the cachyos host went berserk — pegged at ~95-99% CPU each:

| Process | PID (at discovery) | CPU | Notes |
|---------|-------------------|-----|-------|
| `cosmic-osd` | 491358, then 494665 | 95–99% | On-screen display daemon (volume/brightness popups) |
| `cosmic-settings-daemon` | 491350, then 494666 | 90–99% | Settings synchronization daemon |

Both are children of `cosmic-session` (PID 491296), which auto-respawns them on death. Killing them with SIGTERM/SIGKILL just caused immediate respawn into the same broken spin state.

**Fix:** `kill -STOP` (SIGSTOP) freezes the processes without triggering respawn — `cosmic-session` thinks they're still alive. CPU dropped from ~200% combined to idle immediately. Trade-off: no OSD popups (volume/brightness overlays) until next COSMIC session restart or package update.

```bash
# The fix (run as root on cachyos host)
kill -STOP $(pgrep cosmic-osd) $(pgrep cosmic-settings-daemon)
```

Both packages at version `1:1.0.8-1.1`, no updates available via CachyOS repos. Likely a known COSMIC bug where the two processes enter a D-Bus or config-notification feedback loop.

Kill the container, restart it, doesn't matter. The session survives, the files are the same, the tools work. It's like pulling the tablecloth out and the dishes stay put.

### Host-lib mount status

Reviewed the Incus device mounts for glibc/musl coexistence:

| Incus device | Host path | CT mount point | Purpose |
|-------------|-----------|----------------|---------|
| `host-usrlib` | `/usr/lib` | `/host-lib` | Host glibc libs (safe, non-conflicting path) |
| `host-lib64` | `/usr/lib64` | `/usr/lib64` | Host glibc dynamic linker + libs |
| `host-udev` | `/run/udev` | `/opt/host-udev` | Host udev data |

The two library stacks coexist cleanly:

- **Alpine musl** binaries (`/bin/bash`, etc.) → linker `/lib/ld-musl-x86_64.so.1` → searches `/lib`, `/usr/lib` (Alpine's own)
- **Host glibc** binaries (`/usr/sbin/bash`, etc.) → linker `/usr/lib64/ld-linux-x86-64.so.2` → searches `/usr/lib64`, `/usr/lib`

Musl doesn't search `/usr/lib64`, so no cross-contamination. The only shared search path is `/usr/lib`, but inside the CT that's Alpine's own musl-linked libraries — glibc binaries finding musl `.so` files there could theoretically cause issues, but in practice the glibc linker resolves its core deps from `/usr/lib64` first. No problems observed.

### CLAUDE_CODE_SHELL: the fix for bash tool in Alpine CT

The bash tool fails inside the CT because Claude Code bundles its own glibc-linked bash, which can't run on Alpine's musl. The fix is simple — tell Claude Code to use Alpine's native bash:

```json
// ~/.claude/settings.json (inside the CT)
{
  "env": {
    "CLAUDE_CODE_SHELL": "/bin/bash"
  }
}
```

This overrides Claude Code's bundled bash with Alpine's musl-linked `/bin/bash`. Requires a Claude Code restart to take effect.

**If this works**, the `host-lib64` and `host-usrlib` Incus device mounts may no longer be needed at all — they were primarily there to support the glibc bash. Removing them would give a cleaner CT with pure musl, no glibc contamination.

## Session 3: Container Network Migration to br0

### Change

Moved gcwg and mesh3 containers from `incusbr0` (NAT, 10.11.0.x) to `br0` (bridged to LAN, 192.168.2.x) to match cosmixos. All three containers now share the same bridge and are directly reachable on the LAN.

### What was done

1. **Added device-level eth0 overrides** on both containers to use `br0` instead of the default profile's `incusbr0`:
   ```bash
   incus config device add gcwg eth0 nic nictype=bridged parent=br0
   incus config device add mesh3 eth0 nic nictype=bridged parent=br0
   ```

2. **Fixed gcwg static network config** — gcwg had a hardcoded static IP in `/etc/systemd/network/eth0.network` pointing at the old 10.11.0.x network. Changed it to DHCP so it picks up a 192.168.2.x address from the LAN router.

3. **Restarted both containers** to apply.

mesh3 was already using DHCP and picked up the new network automatically.

### Result

| Container | Old network | Old IP | New network | New IP |
|-----------|------------|--------|-------------|--------|
| cosmixos | br0 | 192.168.2.144 | br0 | 192.168.2.144 (unchanged) |
| gcwg | incusbr0 | 10.11.0.2 | br0 | 192.168.2.202 |
| mesh3 | incusbr0 | 10.11.0.14 | br0 | 192.168.2.151 |

### Notes

- The default Incus profile still uses `incusbr0` — these are per-container device overrides, so new containers will still get the NAT network unless explicitly overridden.
- WireGuard mesh IPs (172.16.2.x) are unaffected — those are on the wg0 interface, independent of the bridge change.
- gcwg and mesh3 IPs are now DHCP-assigned and may change on restart. If stable IPs are needed, set DHCP reservations on the router or switch back to static configs with 192.168.2.x addresses.
