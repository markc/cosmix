## Vision Documents

### cosmix-systemd-strategy.md — The Sweet Spot
Identified which systemd components to replace with Rust + cosmix-port and which to keep:

**Replace (high value, achievable):**
- `cosmix-seat` — logind replacement, COSMIC-only, single-seat (~0.5-1 dev-year)
- `cosmix-journal` — journald replacement, dual text+structured logs (~0.5-1 dev-year)
- `cosmix-resolved` — resolved replacement, wraps hickory-dns (~0.3-0.5 dev-year)
- `cosmix-timesync` — timesyncd replacement, trivial NTP client
- `cosmixctl` — unified CLI for all cosmix-port services

**Keep (not worth replacing):**
- systemd PID 1 (15 years of edge cases, must never crash)
- udevd (thousands of vendor rules)
- dbus-daemon (compatibility bus for PipeWire, BlueZ, NetworkManager)
- PipeWire, NetworkManager, polkit (invisible plumbing)

Total effort: ~1.7-3.2 dev-years vs 15-23 for full systemd replacement.

### cosmix-os-vision.md — Alpine + COSMIC + Rust Everywhere

**Core thesis:** Every binary the user directly interacts with is Rust — shell, editor, terminal, file manager, compositor, mail client, system tools. The C lives underneath in the kernel, libc, and system daemons the user never sees.

**Base:** Alpine Linux — musl (true static binaries), apk (fast package manager), no systemd (fill gaps, don't fight incumbents), COSMIC already packaged in edge/community.

**Full Rust userspace audit:** uutils-coreutils, ripgrep, fd, bat, eza, nushell, starship, helix, zellij, alacritty, sudo-rs, ntpd-rs — all available as Alpine packages. Combined with COSMIC desktop (100% Rust) and cosmix services, the user-facing stack is ~95% Rust.

**Deployment variants:** Desktop (Alpine + COSMIC + cosmix), Server (headless + cosmix-daemon + cosmix-web), Embedded/ARM (aarch64, postmarketOS compatible), Custom ISO via mkimage.

### self-improving-build-loop.md — Strix Halo Build Server

**Hardware:** AMD Strix Halo (16C/32T Zen 5, 128GB unified LPDDR5X, RDNA 3.5 40CU) — dual-purpose build farm + LLM inference.

**Host:** Alpine + Incus (no Docker, ever). Containers for:
- Nightly build environment (disposable per-build)
- OpenClaw AI agent (persistent, with vector memory)
- Local Ollama (ROCm GPU passthrough)
- Mattermost (team communication)
- Package repo serving

**The daily cycle:**
1. Daytime: developer codes on cachyos, OpenClaw monitors + tests
2. Evening: full-stack build triggered (all Rust crates, Alpine packages, kernel, ISO)
3. Overnight: automated test (boot ISO in Incus VM, smoke test via cosmix-port)
4. Publish: packages + ISO to cosmix.nexus
5. Morning: dogfood yesterday's build, file bugs, fix, repeat

**Communication:** Mattermost + email (via Stalwart) + cosmix.nexus website. No Signal/Telegram.

**Memory continuity:** Five layers — git (CLAUDE.md), Claude Code auto-memory, OpenClaw file memory, vector embeddings (pgvector), live cosmix-port runtime state. New contributors boot the ISO and have full project context from day one.

**Self-improving:** Each nightly build incorporates the previous day's fixes. OpenClaw learns failure patterns from build logs. Vector memory compounds. Day 365 the AI knows every failure and fix the project has ever seen.

## Research Notes

- **Alpine Linux:** COSMIC fully packaged (22+ packages in edge/community), Incus available, ARM support (aarch64, armv7, riscv64), abuild + mkimage build system
- **COSMIC on Alpine:** Already working — postmarketOS (Alpine-based) drove the musl porting work
- **Rust for Linux kernel:** No longer experimental (Dec 2025 Kernel Maintainer Summit). DRM subsystem will mandate Rust for new GPU drivers.
- **OpenClaw:** Open-source autonomous AI agent (247K GitHub stars). Runs locally, integrates with LLMs, skill-based architecture. Would be modified for Cosmix: native Alpine (no Docker), local Ollama, Mattermost instead of Signal, registered as cosmix-port.
- **musl vs glibc for Rust:** Minimal performance difference — Rust bypasses most libc hot paths (own allocator, string handling, threading). The musl malloc penalty primarily affects C/C++ code.
