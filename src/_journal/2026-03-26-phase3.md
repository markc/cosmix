# 2026-03-26 — Phase 3 complete: cosmix-jmap JMAP server feature-complete

## What was done

### Thread/get and Thread/changes
- New `src/jmap/thread.rs` — Thread/get returns thread objects with `emailIds`
  list, Thread/changes uses changelog-based modseq tracking
- Extended `src/db/thread.rs` — `get_by_ids()` fetches threads with their email
  IDs ordered by received_at, `query_ids()` for pagination

### Email/query expanded filters (RFC 8621 §4.4)
- New `EmailFilter` struct in `db/email.rs` with 15 filter properties:
  `inMailbox`, `inMailboxOtherThan`, `before`, `after`, `from`, `to`, `cc`,
  `subject`, `text` (FTS5 full-text search), `minSize`, `maxSize`,
  `hasAttachment`, `hasKeyword`, `notKeyword`
- Dynamic SQL builder with FTS5 join when `text` filter is used
- Sort support via `isAscending` property in sort array

### Email/import (RFC 8621 §4.6)
- New `import()` method in `jmap/email.rs` — accepts `emails` map with
  `blobId`/`mailboxIds`/`keywords`, reuses `create_from_blob` to parse
  RFC 5322 and store. Used by clients for drag-and-drop import and
  server-to-server message migration.

### EmailSubmission/get and /changes
- New `email_submissions` table — tracks submission records with id, email_id,
  envelope (from/to), send_at, undo_status, delivery_status, queue_id
- `EmailSubmission/set` now persists submission records and records changelog
- `EmailSubmission/get` returns submission objects with full envelope info
- `EmailSubmission/changes` for modseq-based change tracking

### EventSource push (RFC 8620 §7.3)
- `tokio::sync::broadcast` channel in `AppState` for state change notifications
- `GET /jmap/eventsource` — SSE endpoint with query params: `types` (filter
  by object type or `*`), `closeafter` (close after first state event),
  `ping` (keep-alive interval in seconds)
- Streams `StateChange` events with `@type`, `changed` map per RFC 8620
- State changes emitted automatically after any `/set` or `/import` method

### VacationResponse/get and /set (RFC 8621 §8)
- New `vacation_responses` table — singleton per account with isEnabled,
  fromDate, toDate, subject, textBody, htmlBody
- `VacationResponse/get` returns the singleton config
- `VacationResponse/set` patches individual fields with upsert
- Vacation auto-reply hooked into SMTP inbound:
  - Skips spam, bounces (empty sender), and Auto-Submitted messages (RFC 3834)
  - Checks Precedence: bulk/list/junk to prevent mailing list loops
  - Respects fromDate/toDate window
  - Queues reply for outbound SMTP delivery with proper In-Reply-To threading

## Decisions

- **Broadcast channel for push**: `tokio::sync::broadcast` with 256-slot buffer.
  Simple, no external deps. Lagged clients get a debug log and continue.
- **State changes from dispatch**: emitted after `/set` and `/import` methods
  by checking the method name suffix. Clean separation — individual method
  handlers don't need to know about push.
- **Vacation via SMTP queue**: auto-replies go through the normal outbound queue
  rather than direct SMTP send. Gets DKIM signing, retry logic, and delivery
  tracking for free.
- **Bounce already implemented**: discovered that `delivery.rs` already calls
  `generate_bounce()` → `bounce::generate_ndr()` on permanent failure (attempts
  >= 10), delivering the NDR to the sender's Inbox. No work needed.
