## Server Firefighting

### mrn — WordPress cron stampede (load 41)

Root cause: `/usr/local/bin/wp-cron-all` launched all 78 WordPress sites' `wp cron event run` in parallel (`&` in a loop with `wait`) every 15 minutes. Combined with attacker traffic, load hit 41+.

**Fixed:**
- Rewrote `wp-cron-all` to max 3 concurrent processes with 2s stagger between launches
- Load dropped from 41 → 1.3 immediately
- Banned 4 attacker IPs via CrowdSec (wp-login brute force, PHP backdoor scanners)
- Accidentally banned 203.25.132.8 (SpiderWeb Cloud — legitimate IMAP customer with 82 connections), unbanned promptly

### sca — Roundcube webmail slow for India user

User wottapud@spiderweb.com.au in India reporting "too slow to respond" on webmail.spiderweb.com.au (Roundcube on sca VPS).

**Diagnosis:**
- sca itself was fine (load 1.14)
- Roundcube had no IMAP caching enabled — every page load made fresh IMAP queries
- India ↔ Australia latency (~200ms) multiplied by many IMAP round-trips per page

**Fixed:**
- Enabled Roundcube IMAP caching to MariaDB (`imap_cache`, `messages_cache`) with 10-day TTL
- Folder listings and message headers now served from local DB cache
- First load still hits IMAP (cold cache), subsequent views bypass IMAP entirely
- Estimated cache size for ~1,537 users: 50-200MB in MariaDB (trivial)

### Architecture note

For admin@spiderweb.com.au (proxied via imapc to mbx spinning disk), the Roundcube DB cache stacks with the proxy NVMe indexing: proxy handles IMAP-level index lookups on SSD, Roundcube cache eliminates IMAP round-trips entirely for repeat views. When the ~500 spiderweb.com.au mailboxes migrate to the proxy setup (see `~/.ns/_doc/2026-03-04-imapc-index-proxy-plan.md`), webmail users should see comparable performance to direct NVMe VPS mailboxes.
