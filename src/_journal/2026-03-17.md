- **Master key** — single shared secret set in `web.toml` or `COSMIX_MASTER_KEY` env var

## Deployment

Built on cosmixos (Alpine CT, static musl) and deployed to gcwg, mko, mmc.

### ZFS bind mount build gotcha

The cosmix source tree is bind-mounted into the cosmixos CT via ZFS (`zpcachyos/ROOT/cos/home` mounted at `/home/cosmic/.gh/cosmix`). When the host modifies source files, cargo inside the CT doesn't always detect mtime changes — its fingerprinting sees stale timestamps through the idmapped mount.

**Symptom:** `cargo build` says "Compiling cosmix-web" but the output binary is identical to the previous build (same md5, same size). The binary on disk never actually gets rewritten.

**Fix:** Run `cargo clean -p cosmix-web -p cosmix-daemon` before building. This clears fingerprints and forces a real recompile. Costs ~0.5s extra, completely reliable.

**Not needed:** rsync. The bind mount is better — zero copy time, single source of truth. The `cargo clean -p` workaround is baked into `scripts/deploy.sh`.

## Session 4: cosmixos Direct DRM Desktop (no cage)

### The wrong approach (cage / nested Wayland)

The old `scripts/start-cosmixos` used `cage` as a kiosk compositor on the host, with cosmic-comp inside the CT running the **Winit backend** as a nested Wayland client. This was the approach documented in `alpine-cosmic-lxc-guide.md` — useful for running COSMIC-in-a-window during development, but **not** the target architecture.

The old script:
```bash
exec cage -- incus exec cosmixos -- su - cosmic -c '
    export WAYLAND_DISPLAY=wayland-0
    export XDG_RUNTIME_DIR=/mnt/wayland
    exec dbus-run-session -- cosmic-session
'
```

Problems: requires cage on host, requires Wayland socket sharing (`/mnt/wayland`), adds a compositor layer, and doesn't give COSMIC direct GPU access for DRM/KMS.

### The right approach (direct DRM via seatd)

The cosmixos CT already has everything needed for direct rendering:
- **GPU passthrough:** `/dev/dri/card0` + `renderD128` (Incus `gpu` device, gid=1001)
- **Input devices:** `/dev/input/event*` (full keyboard/mouse access)
- **TTYs:** `/dev/tty0`–`/dev/tty7`
- **seatd:** installed, group `seat`, `cosmic` user in `seat` group
- **greetd:** installed (not yet configured)

cosmic-comp detects that no `WAYLAND_DISPLAY` is set and uses the **DRM/KMS backend** — rendering directly to the physical display via the GPU. seatd brokers access to DRM and input devices.

Updated `scripts/start-cosmixos`:
```bash
incus exec cosmixos -- rc-service seatd start
incus exec cosmixos -- rc-service dbus start
incus exec cosmixos -- su - cosmic -c '
    export XDG_RUNTIME_DIR=/run/user/1001
    export XDG_SESSION_TYPE=wayland
    export LIBSEAT_BACKEND=seatd
    exec dbus-run-session -- cosmic-session
'
```

Key differences from the cage approach:
- **No cage, no host compositor involvement** — cosmic-comp IS the compositor
- **No `WAYLAND_DISPLAY`** — its absence tells cosmic-comp to use DRM backend
- **No `/mnt/wayland` mount** — XDG_RUNTIME_DIR is local to the CT
- **`LIBSEAT_BACKEND=seatd`** — explicit seat backend selection
- **seatd must be running** — added to CT's `default` runlevel

### Two valid deployment modes

| Mode | When to use | Backend | Script |
|------|-------------|---------|--------|
| **Direct DRM** | Production desktop, daily driver | DRM/KMS via seatd | `scripts/start-cosmixos` |
| **Nested Wayland** | Dev/testing, COSMIC-in-a-window | Winit via shared socket | `alpine-cosmic-lxc-guide.md` |

The nested approach is still valid for development (run cosmixos as a window while working on the host), but `start-cosmixos` is for the real desktop.

### Current status

- seatd installed and added to default runlevel ✓
- GPU, input, TTY access confirmed ✓
- `start-cosmixos` rewritten for direct DRM ✓
- The "ERROR: dbus has already been started" was harmless — just `rc-service dbus start` finding dbus already running from boot

### Testing results (4 attempts)

**COSMIC desktop renders successfully on the physical display!** cosmic-comp acquires DRM master, the full COSMIC session (panel, wallpaper, launcher) appears on screen. This is the hard part — confirmed working.

**Remaining issue:** keyboard and mouse are unresponsive. The desktop is visible but cannot be interacted with. Reproduced consistently across 4 separate attempts.

The input devices are present in the CT (`/dev/input/event0`–`event20`, `mice`, `mouse0`–`mouse2`) owned by group `hostinput` (GID 994). The `cosmic` user is in the `hostinput` group. Permissions are not the obvious blocker — the issue is likely seatd not granting libinput access to the evdev nodes, or libinput/udev enumeration not finding the devices.

### Root Cause Analysis (Session 6)

Log analysis of `/var/log/cosmic-session.log` revealed two distinct failures:

**Problem 1: seatd VT ioctls fail in container**

```
[ERROR] [common/terminal.c:162] Could not open target tty: Operation not permitted
[ERROR] [seatd/seat.c:70] Could not open tty0 to update VT: Operation not permitted
[ERROR] [seatd/seat.c:557] Could not open VT for client
```

seatd was creating a VT-bound seat (its default). In a container, VT ioctls (`VT_OPENQRY`, `VT_ACTIVATE`, `VT_SETMODE`) fail because the container doesn't own the virtual terminal subsystem — the host kernel does. cosmic-comp connected to seatd, seatd tried to open a VT for it, failed, and cosmic-comp exited with code 1.

**Fix:** `SEATD_VTBOUND=0` environment variable. This is seatd's own mechanism (checked in `server.c:42-45`) to create a non-VT-bound seat. A non-VT-bound seat still brokers DRM master and evdev input device access — it only skips VT switching operations.

**Problem 2: udev can't start — no devtmpfs**

```
daemon.err udev[810]: Udev requires /dev to be a mounted devtmpfs.
```

In an Incus CT, `/dev` is `tmpfs` (managed by the container runtime), not `devtmpfs`. The `rc-service udev start` fails. Without udev running, the udev database has no `ID_INPUT_KEYBOARD`, `ID_INPUT_MOUSE` tags, so libinput's udev-based device enumeration finds nothing — even though `/dev/input/event*` nodes and `/sys/class/input/` symlinks are all present and accessible.

**Fix:** Run `udevd --daemon` manually (it works even on tmpfs `/dev` — it just can't create device nodes), then `udevadm trigger --subsystem-match=input` + `udevadm settle`. This processes the sysfs events for the bind-mounted `/dev/input` devices and populates the udev database. Confirmed working: `udevadm info /dev/input/event3` shows `ID_INPUT_KEYBOARD=1`, `LIBINPUT_DEVICE_GROUP=...` after trigger.

**Also found:** greetd crash-looping due to missing `pam_gnome_keyring.so` — harmless but noisy. Fix: `apk add gnome-keyring` or remove the PAM module reference.

### Updated start-cosmixos Script

Key changes from the previous version:

| Before | After | Why |
|--------|-------|-----|
| `rc-service seatd start` | `seatd-launch -- cosmic-session` | Embedded seatd, no daemon needed |
| `LIBSEAT_BACKEND=seatd` | `SEATD_VTBOUND=0` | Non-VT-bound seat for container |
| `rc-service udev start` (failed) | `udevd --daemon` + `udevadm trigger` | Works on tmpfs /dev |

## Session 7: Input Device Root Cause — udev Tagging Failure

### The problem

libinput discovers input devices via udev database tags (`ID_INPUT_KEYBOARD`, `ID_INPUT_MOUSE`, `LIBINPUT_DEVICE_GROUP`). Without these tags, libinput's udev-based enumeration returns zero devices — the desktop renders but can't receive any input.

### Why udevadm trigger doesn't work in containers

The Session 6 fix (`udevd --daemon` + `udevadm trigger --subsystem-match=input`) was the right idea but **doesn't actually work**. `udevadm trigger` sends synthetic uevents via sysfs to the running udevd daemon. In an Incus CT, udevd silently drops most of these events — only 4-5 of 20+ input devices get processed. The exact reason is unclear (likely related to tmpfs `/dev` or missing cgroup/namespace context), but the result is consistent: most devices never get tagged.

Evidence: after `udevadm trigger`, `udevadm info /dev/input/event6` (USB mouse) showed zero tags. But `udevadm test /devices/.../event6` produced all the correct tags AND wrote them to `/run/udev/data/`.

### First attempt: udevadm test per-device (abandoned)

`udevadm test <sysfs-path>` runs udev rules in-process and writes db entries. This worked but was slow (20+ devices) and fragile. Superseded by the proper fix below.

### Web research — how others solved this

Deep search across Incus/LXD docs, Proxmox forums, Linux Containers forums, sr.ht/seatd, wlroots issues, and libinput source code. Key findings:

**Nobody has done this before.** No public documentation of COSMIC (or any Smithay-based compositor) running inside LXC/Incus with direct DRM rendering. The closest working setups use X11 in LXC, or nested Wayland (not direct DRM). We're in uncharted territory.

**The standard solution for udev in containers:** bind-mount `/run/udev` from the host (read-only). The host's udevd has already tagged every device. libinput's `udev_enumerate_scan_devices()` reads the db files and finds all devices. Confirmed across multiple independent sources (Proxmox forums, Linux Containers forums, Docker/Podman setups).

**Critical sources:**
- [Proxmox: Forward udev properties to container](https://forum.proxmox.com/threads/forward-all-udev-properties-to-container.170247/) — `/run/udev` bind mount
- [CachyOS LXC gist (sway/hyprland)](https://gist.github.com/yvesh/463594f0d3e9174a8032f236a59f8a50) — complete privileged config
- [seatd source: server.c:42](https://git.sr.ht/~kennylevinsen/seatd) — `SEATD_VTBOUND` reads env, creates non-VT-bound seat
- [libinput source: evdev.c:80-94](https://github.com/jadahl/libinput) — full list of required udev properties (ID_INPUT, ID_INPUT_KEYBOARD, ID_INPUT_MOUSE, LIBINPUT_DEVICE_GROUP, etc.)
- [wlroots #2257: Make udev optional](https://github.com/swaywm/wlroots/issues/2257) — libinput REQUIRES libudev, even the path backend
- [libudev-zero](https://github.com/illiliti/libudev-zero) — Alpine package, sysfs-based alternative (fallback option)
- [Toolbox #992](https://github.com/containers/toolbox/issues/992) — bind-mounting `/run/udev/tags` fixes seat matching

**Other tips from research:**
- `lxc.mount.auto = sys:rw` may be needed for DRM master
- `LIBSEAT_BACKEND=builtin` is an alternative to seatd-launch (embedded seatd, needs root)
- Running seatd on the host and passing socket into container is another option
- libinput has NO path to avoid udev entirely — even the "path" backend calls `udev_device_new_from_devnum()`

### The proper fix: bind-mount host /run/udev

LXC mounts tmpfs on `/run` which covers any `lxc.mount.entry` targeting `/run/udev`. Workaround: mount to `/opt/host-udev` via Incus disk device, then bind-mount to `/run/udev` in the start script.

**Incus config:**
```bash
incus config device add cosmixos host-udev disk source=/run/udev path=/opt/host-udev readonly=true
```

**In start-cosmixos:**
```bash
incus exec cosmixos -- sh -c '
    mkdir -p /run/udev
    mount --bind /opt/host-udev /run/udev
'
```

Result: 504 udev db entries visible inside CT. All input devices properly tagged — keyboard, mouse, Lofree, everything. No udevd, no hwdb compilation, no per-device test loop. Just the host's database.

**cosmic-greeter crash-loop:** Disabled from default runlevel (missing `pam_gnome_keyring.so`).

**strace finding:** cosmic-comp SIGABRT when `/run/user/1001/` missing → XDG_RUNTIME_DIR must be created before launch.

- `event10`: Compx Flow84@Lofree (consumer) → `ID_INPUT_MOUSE=1`, `LIBINPUT_DEVICE_GROUP`

## Session 8: seatd-launch Strips Environment — The Real VT-Bound Fix

### The problem

Sessions 4–7 all set `SEATD_VTBOUND=0` to create a non-VT-bound seat (required for containers where VT ioctls fail). The log consistently showed:

```
[INFO] [seatd/seat.c:48] Created VT-bound seat seat0
[ERROR] [seatd/seat.c:557] Could not open VT for client
```

cosmic-comp exited with code 1. The desktop never started (or if it did via a race, had no input).

### Root cause: seatd-launch passes empty environment

`seatd-launch` (seatd 0.9.3, Alpine Edge) explicitly passes a **NULL environment array** to the seatd child process:

```c
// seatd-launch/seatd-launch.c
char *env[1] = {NULL};
execve(SEATD_INSTALLPATH, command, env);
```

This means **every environment variable is stripped** before seatd starts — `SEATD_VTBOUND=0` never reaches the seatd process, regardless of how it's set in the parent shell.

### Evidence

```bash
# Via seatd-launch — env var stripped, VT-bound:
SEATD_VTBOUND=0 seatd-launch -- echo test
# → "Created VT-bound seat seat0"

# Direct seatd — env var preserved, non-VT-bound:
SEATD_VTBOUND=0 seatd -l debug &
# → "Created seat seat0"  (no "VT-bound")
```

The `SEATD_VTBOUND` env var is read in `seatd/server.c:42-45` and has been present since seatd 0.8.0. The code works — `seatd-launch` just prevents it from ever being seen.

### Fix

Replaced `seatd-launch -- cosmic-session` with direct seatd daemon + `LIBSEAT_BACKEND=seatd`:

```bash
# Start seatd as daemon with non-VT-bound seat
SEATD_VTBOUND=0 seatd -u cosmic -g seat -l debug &

# cosmic-session connects to daemon socket
su - cosmic -c '
    export LIBSEAT_BACKEND=seatd
    exec dbus-run-session -- cosmic-session
'
```

### Changes

| Target | Change |
|--------|--------|
| `scripts/start-cosmixos` | Replaced `seatd-launch` with direct `seatd` daemon + `LIBSEAT_BACKEND=seatd` |
| `scripts/start-cosmixos` | Added seatd cleanup on session exit (`pkill -x seatd`) |

## Session 9: Input Device Fix — cgroup Allow + disown Bug

### The real input blocker: cgroup device permissions

Analysis of `/var/log/start-cosmixos.log` confirmed the Session 8 seatd fix worked — direct seatd daemon with `SEATD_VTBOUND=0` successfully created a non-VT-bound seat, cosmic-comp acquired DRM master, and the full COSMIC session rendered on screen.

However, `libinput list-devices` showed every evdev device failing with "Operation not permitted" — even as root. The `/dev/input` bind mount (`raw.lxc: lxc.mount.entry`) made the device nodes visible, but the **cgroup v2 BPF device controller** blocked access because character device major 13 (input) was never whitelisted.

**Fix:** Added `lxc.cgroup2.devices.allow = c 13:* rwm` to the Incus CT config:

```bash
incus config set cosmixos raw.lxc "lxc.mount.entry = /dev/input dev/input none bind,create=dir 0 0
lxc.cgroup2.devices.allow = c 13:* rwm"
```

After CT restart, `libinput list-devices` found all devices: AT keyboard, USB optical mouse, Lofree Flow84 (keyboard + pointer), Audient iD4.

### The disown bug

The first run (18:19:01, `set -e`) failed with `Error: Command not found` at the seatd startup line. This was Incus's error format (not bash's), which was confusing.

**Root cause:** `ct_sh` runs `incus exec cosmixos -- sh -c "..."` — Alpine's `sh` is busybox ash, which does not have `disown` (a bash builtin). The command `setsid ... & disown` failed because ash couldn't find `disown`. With `set -e`, the script exited immediately.

The second run (18:20:42) worked because `set -e` had been changed to `set -x` (debug tracing), so the `disown` failure was ignored and seatd started successfully via the `&` backgrounding.

**Fix:** Removed `disown` from the seatd startup line — `setsid` already creates a new session, making `disown` redundant. Restored `set -ex` (both error exit and tracing).
