# 2026-04-01 — UI Foundation: dx-components, Tailwind v4, cosmix-preview

## The Problem

Every cosmix app used inline `style:` strings, rediscovered the same WebKitGTK workarounds independently, and had no shared components. Yesterday's 2-hour toggle button fight proved this was unsustainable.

## What Was Built

### dioxus-primitives + dx-components integrated into cosmix-lib-ui

- **dioxus-primitives** (git dep) — 34 unstyled accessible behavioural primitives (ARIA, keyboard nav, focus management)
- **dx styled components** — all 40 official DioxusLabs components added via `dx components add`, moved into `cosmix-lib-ui/src/dx_components/` for shared access
- **lucide-dioxus** v2.577 — 1000+ icon components, re-exported via `cosmix_ui::icons::lucide::*`

Components available: accordion, alert_dialog, aspect_ratio, avatar, badge, button, calendar, card, checkbox, collapsible, context_menu, date_picker, dialog, drag_and_drop_list, dropdown_menu, form, hover_card, input, label, menubar, navbar, pagination, popover, progress, radio_group, scroll_area, select, separator, sheet, sidebar, skeleton, slider, switch, tabs, textarea, toast, toggle, toggle_group, toolbar, tooltip, virtual_list

### Theme bridge in theme.rs

The dx-components use their own CSS variable system (`--primary-color-*`, `--secondary-color-*`, `--sidebar-*`). Instead of loading their `dx-components-theme.css` (which has its own dark/light mechanism that fights our theme), we output concrete OKLCH values for ALL their vars directly from `generate_css()`.

This means: one `use_theme_css()` call, light/dark toggle works instantly for both our components AND all dx-components, no CSS variable resolution chains, no FOUC.

### Tailwind CSS v4

- Dioxus 0.7 auto-detects `tailwind.css` at crate root, runs Tailwind watcher via `dx serve`
- `@theme inline` maps our OKLCH custom properties to Tailwind utility classes (`bg-bg-secondary`, `text-fg-muted`, `text-danger`, etc.)
- Per-app `tailwind.css` only contains class mapping — bridge vars live in theme.rs
- Apps must use `dx serve` (not `cargo run`) for Tailwind classes to compile

### Standard app shell pattern established

Every cosmix app must follow:
```
MenuBar (always on top, with File > Quit and Help > About minimum)
Content area (flex-1 overflow-y-auto, may contain Sidebar, Tabs, etc.)
```

This was codified in CLAUDE.md as a priority.

## Key Decisions

- **dx-components-theme.css is NOT used** — fully replaced by bridge vars in theme.rs. Their dark/light mechanism (CSS `--dark`/`--light` toggle) conflicted with our OKLCH theme.
- **Bridge vars use concrete values, not var() references** — `--primary-color: oklch(12% 0.015 220.0)` not `--primary-color: var(--bg-primary)`. This eliminates CSS variable resolution issues and makes light/dark instant.
- **cosmix-lib-ui is THE priority crate** — all shared UI components, theme logic, menu system, and layout patterns must live here. Never duplicate across apps.
- **`dx serve` required for Tailwind** — `cargo build` apps don't get Tailwind class compilation. Migrate all GUI apps to `dx serve` over time.
- **Sidebar demo is the premier component** — full upstream example adapted with OKLCH theme. This is the precursor for cosmix-shell.
