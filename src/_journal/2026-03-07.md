# 2026-03-07 — Rename to Cosmix + Layered Architecture

## Rename

Project renamed from **Coworker** to **Cosmix** (COSMIC + remix/mix). Better name — signals the COSMIC ecosystem, memorable, distinctive.

## Architecture Revision

Replaced the linear phase plan with a **three-layer architecture**:

### Layer 1: External Daemon (zero app changes)
- AT-SPI2 for UI introspection
- EIS/libei for input injection
- ext-foreign-toplevel-list for window management
- D-Bus for existing interfaces
- AutoHotkey/Hammerspoon model

### Layer 2: Compositor Scripting (cosmic-comp only)
- Embed mlua in cosmic-comp (already being patched)
- Lua window rules, keybindings, workspace management
- Awesome WM model

### Layer 3: libcosmic Integration (the endgame)
- `cosmix-port` crate — IPC listener + command registry
- Feature-gated in libcosmic, minimal per-app code
- Apps register commands, daemon routes Lua calls
- The ARexx model: apps link the library, expose a port

### Key decisions
- **Unix socket** for per-app IPC (not D-Bus, not WebSocket)
- **Socket discovery** via `/run/user/$UID/cosmix/` directory
- **Lua stays in daemon only** — per-app crate is just IPC + registry
- **WebSocket** reserved for cross-node mesh (meshd)
- **D-Bus** consumed (existing interfaces) but not built upon

### Transport rationale
D-Bus is too complex for building new automation interfaces. Unix sockets are fast, simple, and auto-discoverable via the filesystem. WebSocket is for the meshd cross-node layer only.
