# 2026-03-21 — Mesh Network Fixes, COSMIC OSD Spin Fix, Nightly Build Strategy

## Session 1: Infrastructure Fixes + Build Planning

### WireGuard wg1 Fix

**Problem:** `wg-quick@wg1` failed on every boot since the switch from `incusbr0` to `br0` (bridged networking for Incus CTs on 192.168.2.0/24). The PostUp/PostDown commands in `/etc/wireguard/wg1.conf` still referenced `incusbr0`, which no longer exists at boot.

**Fix:** `sed -i 's/incusbr0/br0/g' /etc/wireguard/wg1.conf`

**Result:** wg1 starts cleanly, all mesh nodes reachable:
- mko (172.16.2.210) — 1.8ms
- gcwg (172.16.2.4) — 1.6ms
- mmc (172.16.2.9) — 30ms

### sshd Boot Failure

**Problem:** sshd is enabled but didn't survive boot — started for 29ms then died (Type=notify-reload, so systemd treated it as a failed start and didn't auto-restart). Journal entries lost to rotation.

**Fix:** Added drop-in `/etc/systemd/system/sshd.service.d/restart.conf`:
- `After=network-online.target`
- `RestartSec=5`, `StartLimitIntervalSec=60`, `StartLimitBurst=5`

### cosmic-osd 100% CPU Spin — Root Cause Found and Fixed

**Root cause chain:**
1. `cosmic-settings-subscriptions` crate (used by both `cosmic-settings-daemon` and `cosmic-osd`) has `pa_mainloop_iterate(false)` — non-blocking
2. When `pipewire-pulse` isn't ready at boot, `iterate(false)` returns immediately → tight busy-spin → 100% CPU
3. `cosmic-settings-daemon` spins → cascades into `cosmic-osd` spinning (retrying volume/device notifications)
4. Firefox and Thunderbird become unresponsive (Wayland compositor event loop starved)

**Three separate repos have the same bug:**

| Repo | Used by | Status |
|------|---------|--------|
| `pop-os/cosmic-settings` (`subscriptions/pulse/src/lib.rs`) | cosmic-osd | PR #1919 filed (not yet merged) |
| `pop-os/cosmic-settings-subscriptions` (`src/pulse.rs`) | cosmic-settings-daemon | **Fixed locally today** |
| `pop-os/cosmic-settings` (`subscriptions/sound/src/pulse.rs`) | cosmic-osd (newer branch) | Same pattern, not yet fixed |

**Fix applied:** Rebuilt `cosmic-settings-daemon` 1.0.8 from CachyOS PKGBUILD with patched `cosmic-settings-subscriptions` checkout:
- `iterate(false)` → `iterate(true)` (blocking)
- Added `State::Failed | State::Terminated` handling (was only checking `State::Ready`, so failed connections looped forever)

**Build approach:** Modified CachyOS PKGBUILD `prepare()` to sed-patch the cargo git checkout after `cargo fetch --locked`. Had to also clear fingerprint cache to force recompilation (`rm -rf target/release/.fingerprint/*cosmic_settings_subscriptions*`).

**Installed:** `sudo pacman -U cosmic-settings-daemon-1:1.0.8-1-x86_64.pkg.tar.zst`

**Verification:** `strings binary | grep "PulseAudio context connection failed"` confirms fix in binary. Both daemon and OSD at 0% CPU after restart.

**Note:** Attempted to also rebuild `cosmic-osd` with the same fix but hit `libspa-sys 0.8.0` vs PipeWire 1.6.2 incompatibility (struct layout change in `spa_pod_builder` — bindgen generates opaque struct). Would need `pipewire-rs 0.9` but that's a breaking API change. Deferred.

### COSMIC Nightly Build Strategy

**Discovery:** CachyOS already provides two tiers of COSMIC packages:
- `cachyos-extra-v3`: Release packages (epoch-1.0.8), x86_64-v3 optimised
- `cachyos`: `-git` packages tracking COSMIC master (23 packages, rebuilt ~weekly)

The `-git` packages already handle Arch compatibility (newer PipeWire, systemd, glibc vs Pop!_OS base).

**Chosen strategy: Option 2 — CachyOS `-git` as base + selective local rebuilds**

Use CachyOS `-git` packages for the ~20 packages we don't patch. Only build locally the 3-5 packages with cosmix patches:
- `cosmic-comp-git` (animation timings from `overrides.toml`)
- `cosmic-settings-daemon-git` (pulse spin fix)
- `cosmic-osd-git` (pulse spin fix, once libspa-sys resolved)
- `cosmix-daemon`, `cosmix-web` (our own crates)

**Build infrastructure:** Incus CT based on `images:archlinux` with CachyOS repos bootstrapped:
1. Install `cachyos-keyring`, `cachyos-mirrorlist`, `cachyos-v3-mirrorlist` from cdn77
2. Add CachyOS repo sections to `/etc/pacman.conf`
3. `pacman -Syu` + install rust/cargo + build deps
4. Build patched packages, host in local pacman repo on cachyos host
5. Host repo at `/srv/cosmic-nightly/`, priority above `cachyos-extra-v3`

**x86_64-v3 confirmed:** Host CPU supports v3 (and v4), CT shares kernel so `-v3` binaries work.

## Session 2: Build CT Setup — Complete

### Incus Build CT (`cosmix-build`)

Created `images:archlinux` container and bootstrapped CachyOS repos. Key lesson: **CachyOS's patched pacman must be installed first** — vanilla Arch pacman doesn't understand `x86_64_v3` architecture, so v3 repo packages all fail with "does not have a valid architecture". Bootstrap order:

1. Install `cachyos-keyring`, `cachyos-mirrorlist`, `cachyos-v3-mirrorlist` from cdn77 URLs
2. Add only `[cachyos]` repo (non-v3) to pacman.conf
3. `pacman -S cachyos/pacman` — replaces upstream pacman with CachyOS fork that understands v3
4. Now add `[cachyos-v3]`, `[cachyos-core-v3]`, `[cachyos-extra-v3]` repos
5. `pacman -Syu` — full system upgrade pulls v3 optimised packages

**CT configuration:**
- IP: 192.168.2.118 (br0 DHCP)
- Rust: 1.94.0 stable (rustup)
- Linkers: mold + lld
- Build deps: libpulse, pipewire, wayland, libxkbcommon, libinput, systemd, cmake, clang, pkg-config
- Build user: `builder` with passwordless sudo (makepkg can't run as root)
- Snapshot: `base-setup` — clean rollback point

### Test Build: `cosmic-settings-daemon-git` with Pulse Fix

Built from AUR PKGBUILD (`aur.archlinux.org/cosmic-settings-daemon-git.git`), not the CachyOS release PKGBUILD. The AUR `-git` PKGBUILD clones master and uses `mold` linker.

**Patch injection:** Added sed commands in `prepare()` after `cargo fetch --locked` to patch `~/.cargo/git/checkouts/cosmic-settings-subscriptions-*/src/pulse.rs`:
- `iterate(false)` → `iterate(true)`
- State check: `== State::Ready` → match with `Failed|Terminated` handling

**Also changed in PKGBUILD:**

**Result:** Built in 1m31s. Fix verified: `strings binary | grep "PulseAudio context connection failed"` — present.

### Build Workflow (Validated)

```bash
# 1. Clone AUR -git PKGBUILD in CT
incus exec cosmix-build -- su - builder -c 'git clone https://aur.archlinux.org/cosmic-foo-git.git'
# 2. Apply patches to PKGBUILD prepare(), build
incus exec cosmix-build -- su - builder -c 'cd cosmic-foo-git && makepkg -sf --noconfirm'
# 3. Pull package to host repo
incus file pull cosmix-build/tmp/.../foo.pkg.tar.zst /srv/cosmic-nightly/
# 4. Update repo DB
repo-add /srv/cosmic-nightly/cosmic-nightly.db.tar.zst /srv/cosmic-nightly/*.pkg.tar.zst
# 5. Install on host
sudo pacman -Syu cosmic-foo-git
```

## Session 3: Arch Desktop CT + Greeter Integration

### `cosmix-desktop` CT Created

Built a second Incus CT from `images:archlinux` for testing the COSMIC desktop with glibc/CachyOS packages, matching the Alpine cosmixos CT's device config exactly:

**Incus config (mirrors cosmixos):**
- `security.privileged: true` + `raw.idmap: both 1001 1001`
- GPU: `gpu` device with `gid=1001`
- Input: `lxc.mount.entry = /dev/input` + `lxc.cgroup2.devices.allow = c 13:* rwm`
- Audio: `lxc.mount.entry = /dev/snd` + `lxc.cgroup2.devices.allow = c 116:* rwm`
- Home: `/home/cosmix` bind-mount with `shift=true`, `propagation=shared`
- Host udev: `/run/udev` → `/opt/host-udev` (readonly)
- Claude Code: `/opt/claude-code` passthrough

**Key difference from Alpine:** systemd-based, so `ct_sh` uses `bash` not `sh`, and dbus/seatd management uses `systemctl` instead of `rc-service`.

**CachyOS COSMIC packages pulled `mesa-git`** which conflicts with release `vulkan-intel`. Used `vulkan-intel-git` instead. CachyOS's `-git` ecosystem is self-consistent — mixing release and -git mesa packages causes conflicts.

**User:** `cosmix` UID 1001, groups: wheel, video, render, input, tty, seat.

**Snapshot:** `snap2` (ZFS snapshots kept silently failing when CT was running with pending privileged mode changes — had to stop CT first).

### Greeter Session Integration

- Uses `bash` and `systemctl` instead of `sh` and `rc-service`

**Three sessions now available at greeter:**

|---------|-------------|
| COSMIC | Host desktop (native) |
| CosmixOS Alpine | Alpine/musl CT (existing) |
| CosmixOS Arch | Arch/CachyOS/glibc CT (new) |

### musl vs glibc Desktop Performance Analysis

**Why glibc CT should be faster for GUI:**
- iced/libcosmic rendering: heavy allocation in layout/render passes — glibc's per-thread arena malloc vs musl's global-lock allocator
- Font rendering (fontconfig + freetype + harfbuzz): many small allocations, musl's worst case
- Mesa Intel Xe driver: constant command buffer alloc/free, glibc tcache makes this near-free
- CachyOS x86_64-v3 compiled packages add AVX2 optimisation throughout

**For headless daemons (cosmix-daemon):** negligible difference. Rust binary needs only glibc 2.34, Debian Trixie has 2.41 — glibc builds deploy everywhere we need them.

**Decision:** Abandon Alpine/musl static build path. One glibc build deploys to CachyOS host, Debian Trixie nodes (mko, mmc), and Arch CTs. Simpler toolchain, better desktop performance, no practical runtime difference for daemons.

### Package Hold: Preventing Update Overwrites

Regular `pacman -Syu` replaced our patched `cosmic-settings-daemon` with the upstream release. Fixed with:

```
# /etc/pacman.conf
IgnorePkg = cosmic-settings-daemon cosmic-settings-daemon-git
```

Reinstalled patched `-git` version from `/srv/cosmic-nightly/`. Both package names held to prevent either release or CachyOS `-git` updates from overwriting. Remove from `IgnorePkg` when upstream merges the pulse fix (PR #1919).

## Session 4: Architecture Pivot — Dioxus Fullstack

### Decision: Pause libcosmic, Rebirth on Dioxus

After three sessions of fighting COSMIC upstream friction (ignored cosmic-comp PRs, pulse spin bug across 3 repos, libspa-sys/pipewire-rs version incompatibilities, musl vs glibc build matrix), the cumulative cost of maintaining compatibility with Pop!_OS-targeted COSMIC packages on Arch has become the single biggest impediment to progress.

**The trigger:** Realising that the web UI (Axum+HTMX+Lua) and desktop apps (libcosmic) are completely separate codebases solving the same problem — showing UI and calling backend services. Dioxus unifies them.

### What Dioxus Gives Us

**One codebase, two targets:**
- Desktop: Wry webview (same engine as Tauri) — runs as a native window
- Web: WASM in browser — same components, same styling
- Server functions: backend Rust callable directly from UI, same binary

**Tailwind CSS:** Full stylistic control over every pixel. No libcosmic theming constraints. Dark mode, responsive design, the whole ecosystem.

**WebSocket as universal transport:**
- Local IPC: `ws://localhost` (replaces Unix socket cosmix-port)
- Mesh: `wss://node.mesh` (already using WebSocket)
- Dioxus server functions: built-in (replaces HTTP API)
- One protocol everywhere — local and remote are the same code path

### What Stays

| Component | Status |
|-----------|--------|
| Rust + Lua (mlua) | Stays — Lua for automation scripting |
| PostgreSQL brain | Stays — every node has PG |
| WireGuard mesh | Stays — transport layer unchanged |
| WebSocket mesh transport | Stays — becomes THE universal transport |
| ARexx port model | Stays — ports register over WebSocket instead of Unix socket |
| cosmix name | Stays |
| cosmix-daemon (mesh, WG, PG, Lua) | Stays — backend logic reusable |

### What Changes

| Before | After |
|--------|-------|
| libcosmic GUI apps (cosmix-calc, -view, -mail) | Dioxus desktop (webview + Tailwind) |
| Axum + HTMX + Lua web templates | Dioxus fullstack (server functions) |
| Unix socket IPC (cosmix-port) | WebSocket port registry |
| Layer 2/3 compositor scripting | Dropped — WebSocket orchestration instead |
| Separate cosmix-web binary | Merged into single Dioxus fullstack binary |
| COSMIC theme/header integration | Not needed — Tailwind handles all styling |

### What We Replace (Non-Browser Apps)

Everything that isn't Firefox or Thunderbird becomes a Dioxus app:
- Mail client (cosmix-mail)
- File manager
- Calculator
- Text editor / viewer
- System settings
- Terminal (stretch goal)

### Risk Assessment

- **Dioxus maturity:** 0.6 is current (Dec 2024), 0.7 in development. Webview renderer is production-stable (Wry/Tauri proven). WGPU renderer is experimental but not needed.
- **VC-funded:** Full-time developers, consistent releases every 6-9 months, 25k+ GitHub stars
- **Webview performance:** Slight overhead vs native, but with Tailwind + modern CSS the UX can be better than libcosmic
- **No COSMIC integration:** Desktop apps are "web pages in a window" — but with full system API access via Rust
